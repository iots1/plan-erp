01 — Overview & Principles

ภาพรวม & หลักการ

เอกสารนี้สรุป convention กลางขององค์กร สำหรับทุกบริการ NestJS — ตั้งแต่การตั้งชื่อไฟล์/คลาส/ตาราง ไปจนถึงรูปแบบ Response, การ throw error และ Query Params เป้าหมายคือความ สม่ำเสมอ ทำให้ทั้ง codebase และ schema เข้าใจตรงกัน อ่านโค้ดแล้วรู้เจตนาทันที

NAMING

Singular Module · Plural Controller/Service · snake_case property/column · kebab-case ไฟล์/โฟลเดอร์

RESPONSE

JSON:API envelope: status · data|errors · meta · links — ห่ออัตโนมัติด้วย TransformInterceptor

ERRORS

Exception เฉพาะเจาะจงเสมอ · code 6 หลัก · controller ไม่ catch ปล่อยขึ้น global filter

CORE PRINCIPLE

โครงสร้าง envelope ชัดเจน · ส่ง อย่างใดอย่างหนึ่งระหว่าง data กับ errors ไม่ปนกัน · ใช้ exception ที่เจาะจงแทน Error ทั่วไป · ใส่ context เต็มใน exception · ไม่ leak ข้อมูล sensitive (password/token/internal) · log ทุกอย่างภายใน

02 — NestJS Naming Convention

การตั้งชื่อ (NestJS)

Module เป็น เอกพจน์ (แคปซูลเดียวของฟีเจอร์) ส่วน Controller/Service เป็น พหูพจน์ (จัดการหลาย endpoint/method) · Property ของ Entity/DTO เป็น snake_case เพื่อ map 1-to-1 กับคอลัมน์ PostgreSQL ลดการใช้ @Column({ name: '…' })

CategoryPatternExamples
ModuleSingular + ModuleGoodsReceiptModule, MedicalVisitModule
ControllerPlural + ControllerGoodsReceiptsController, PatientsController
ServicePlural + ServiceGoodsReceiptsService, PatientsService
Entity / DTO (class)Singular · PascalCaseGoodsReceipt, CreateGoodsReceiptDTO
Propertysnake_caseimport_cost, lot_number, is_active
EnumSingular · PascalCaseGoodsReceiptStatus, QueueStatus
Folderkebab-casegoods-receipt, patient-change-request
Filenamekebab-case + .type.tsgoods-receipt.module.ts, goods-receipts.controller.ts
ตัวอย่าง · Entity property = snake_case (1-to-1 กับคอลัมน์)
@Entity('goods_receipts')
export class GoodsReceipt {
  @PrimaryGeneratedColumn('uuid')  id: string;
  // snake_case → ไม่ต้องใส่ @Column({ name: '...' })
  @Column()  po_id: string;
  @Column()  supplier_id: string;
  @Column({ type: 'numeric' })  import_cost: number;
  @Column()  lot_number: string;
  @Column({ type: 'date', nullable: true })  expiry_date: Date;
  @Column({ default: false })  is_cancelled: boolean;
}
Class vs Property

คลาสเป็น PascalCase เอกพจน์ (มาตรฐาน OOP/NestJS) แต่ property เป็น snake_case (มาตรฐาน PostgreSQL) — สองระดับนี้ต่างกันโดยตั้งใจ

03 — Boolean & Status Prefixes

Prefix ของ Boolean / Status

ฟิลด์ boolean/status ต้องมี prefix เสมอ เพื่อไม่ให้กำกวมระหว่าง “คำนาม / คำกริยา / สถานะ” — เช่น active อาจเป็นคำนามหรือคำสั่ง แต่ is_active ตอบชัดว่า “มันกำลัง active หรือไม่?”

PrefixUse CaseความหมายExamples
is_Status / Identity“อยู่ในสถานะ/ชนิดนี้ไหม?”is_active, is_deleted, is_verified, is_public
has_Possession / Existence“มี/ประกอบด้วยไหม?”has_attachment, has_permission, has_signature
can_Ability / Rights“ทำได้/ได้รับอนุญาตไหม?”can_edit, can_delete, can_approve
should_Conditions / Actions“ควรเกิดสิ่งนี้ไหม?”should_notify, should_archive, should_refresh
❌ กำกวม
approved: boolean;
email_verified: boolean;
admin: boolean;
public: boolean;
✅ ชัดเจน
is_approved: boolean;
is_email_verified: boolean;
is_admin: boolean;
is_public: boolean;
04 — PostgreSQL Naming

การตั้งชื่อ Database

ตารางเป็น พหูพจน์ snake_case (สื่อว่าเป็น collection ของ record) · คอลัมน์ snake_case · และตั้งชื่อ constraint/index แบบมี prefix ให้ค้นหาและสื่อความหมายได้ทันที

ComponentFormatExamples
Tableplural · snake_casegoods_receipts, lots, patient_insurances
Columnsnake_caseimport_cost, birth_date, is_active
Primary keyid · pk_<table>id, pk_goods_receipts
Foreign keysingular_table_idproduct_id, patient_id, warehouse_id
Indexidx_<table>_<cols>idx_lots_expiry_date, idx_patients_national_id
Unique constraintuq_<table>_<cols>uq_lots_lot_number, uq_users_email
Check constraintchk_<table>_<condition>chk_lots_available_qty, chk_visits_visit_type
ตัวอย่าง · DDL ตาม convention
CREATE TABLE goods_receipts (
  id            uuid DEFAULT gen_random_uuid(),
  po_id         uuid NOT NULL,          -- FK → supplier-bc (ข้าม BC = UUID เท่านั้น)
  warehouse_id  uuid NOT NULL,
  status        varchar NOT NULL,
  is_cancelled  boolean NOT NULL DEFAULT false,
  CONSTRAINT pk_goods_receipts PRIMARY KEY (id),
  CONSTRAINT chk_goods_receipts_status CHECK (status IN ('DRAFT','SUBMITTED','CANCELLED'))
);
CREATE INDEX idx_goods_receipts_po_id ON goods_receipts (po_id);
05 — API Response (JSON:API)

โครงสร้าง Response

ทุก Response ห่อด้วย envelope แบบ JSON:API: status · data หรือ errors (อย่างใดอย่างหนึ่ง) · meta · links — ทำงานอัตโนมัติผ่าน TransformInterceptor เมื่อ controller ประกาศ @ResourceType()

โครง envelope · Success
{
  "status": { "code": 200000, "message": "Request Succeeded" },
  "data": {
    "type": "goods-receipts",          // จาก @ResourceType('goods-receipts')
    "id": "uuid-grn",
    "attributes": {                    // snake_case + i18n nested object
      "status": "SUBMITTED",
      "po_id": "uuid-po", "import_cost": 98.5,
      "remark": { "th": "ของครบ", "en": null },
      "product": { "name": { "th": "ยาเซโกร", "en": "Zagro" } }
    }
  },
  "meta": { "timestamp": "2026-07-14T10:30:00.000Z" },
  "links": { "self": "/inventory/goods-receipts/uuid-grn" }
}
SINGLE

data: { type, id, attributes }

COLLECTION

data: [ { type, id, … } ]

PAGINATED

data: [ … ] + meta.pagination

@ResourceType จำเป็นบนทุก Controller
@Controller('goods-receipts')
@ResourceType('goods-receipts')   // ✅ กำหนด type ใน response · ถ้าไม่ใส่ = คืน raw ไม่ transform
export class GoodsReceiptsController {
  @Get()  findAll() { return this.service.findAll(); }
}
STATUS CODE · 6 หลัก

status.code เป็นเลข 6 หลัก = HTTP status × 1000 + serial · success = 200000 · ต้องมี อย่างใดอย่างหนึ่งระหว่าง data กับ errors ไม่ส่งพร้อมกัน

i18n interplay

ภายใน attributes ฟิลด์ 2 ภาษาถูกยุบเป็น nested object { th, en } ด้วย LocalizationInterceptor เสมอ (แม้ค่าเป็น null) — ห้าม frontend ต่อ string key เอง ดู i18n Guide

CONTROLLER · ไม่ catch

controller ปล่อย exception ลอยขึ้น AllExceptionsFilter — catch เฉพาะกรณีต้องเพิ่ม context เฉพาะทาง (เช่น file upload) เท่านั้น

NULL CHECK · หลัง query

เช็ค null หลัง DB read เสมอ → throw new NotFoundException(...) · ห้าม service คืน null

SECURITY · ไม่ leak internal

ห้ามส่ง stack/token/ข้อความภายในออกไป · ใช้ข้อความ generic + log context เต็มภายใน (service, userId, ids)

07 — Query Parameters

Query Params

ใช้ @ValidatedQuery(QueryParamsDTO) แทน @Query() — ได้ทั้ง type-safe และ throw error code ถูกต้อง (400002) · ถ้าใช้ @Query() กับ DTO ที่มี type จะไปโดน Global ValidationPipe → ได้ 400001 ที่ผิด

❌ @Query()
@Get()
findAll(@Query() q: any) {
  // เสีย type safety
  return this.service.findAll(q);
}
✅ @ValidatedQuery
@Get()
findAll(@ValidatedQuery(QueryParamsDTO) q: QueryParamsDTO) {
  // type-safe + throw 400002
  return this.service.findAll(q);
}
Feature@Query()@Query() + type@ValidatedQuery(DTO)
Type Safety❌ (any)⚠️ (โดน Global Pipe)✅ Full
Error Code—400001 (ผิด)400002 (ถูก)
ValidationManualAuto (pipe ผิด)Auto (ถูก)
Recommended❌❌✅
FEATURES · @ValidatedQuery

แปลง type อัตโนมัติ ("123"→123, "true"→true) · strip property แปลกปลอม (whitelist) · รองรับ nested validation และ array filter syntax filter[]=field||$eq||value

Manual throw · custom validation (query)
import { InvalidParameterException } from '@lib/common/utils/http-exception/invalid-parameter.exception';

@Get('search')
async search(@Query('keyword') keyword: string) {
  const allowed = ['lot_number', 'product_id'];
  const field = keyword?.split(':')[0];
  if (!allowed.includes(field)) {
    throw new InvalidParameterException([{
      field: 'keyword',
      message: `Invalid search field '${field}'. Allowed: ${allowed.join(', ')}`,
    }]);
  }
  return this.service.search(keyword);
}
