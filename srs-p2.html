01

Business Overview & Roles · ใครตั้งค่าอะไร

MASTER DATA ADMIN
ผู้ดูแลข้อมูลหลัก

ตั้งค่า UOM, Item Group (โครงต้นไม้), Brand และ Warehouse — โครงสร้างพื้นฐานที่ทุกสินค้าอ้างถึง

PRODUCT OWNER
ผู้ดูแลสินค้า

สร้าง Item master (2 ภาษา), กำหนด stock UOM + conversion, สร้าง variant จาก attribute และตั้งจุดสั่งซื้อ

PURCHASING
ฝ่ายจัดซื้อ

ตั้งค่า Supplier (ใน supplier-bc) — inventory อ้างถึงด้วย supplier_id (UUID) ตอนรับสินค้า

วัตถุประสงค์ของเฟส: master data คือ รากฐาน — ต้องตั้งค่าให้เสร็จก่อนเปิดทำธุรกรรมใด ๆ (Goods Receipt / PO / SO) · ทุกฟิลด์ 2 ภาษาเก็บเป็น flat column name_th / name_en และทุกตารางสืบทอด BaseEntity (audit + soft-delete) ตาม Backend Conventions

03

Setup Sequence & Handoff to Inventory · ลำดับตั้งค่า → ส่งต่อ

ต้องตั้งค่าตาม ลำดับ dependency (อันหลังอ้างอิงอันหน้า) จึงจะเริ่มรับสินค้า (Goods Receipt · Phase 3) ได้

1

UOM — ตั้งหน่วยนับ + ระบุ is_whole_number (seed ชุดมาตรฐาน)

2

Item Group (tree) + Brand — โครงหมวดหมู่และยี่ห้อ

3

(ถ้ามีตัวเลือก) Item Attribute + values — เตรียมก่อน generate variant

4

Product (Item master) — เลือก item_group + stock_uom (+ brand/supplier) · ตั้ง has_variants ถ้าต้องการ generate

5

UOM Conversion ต่อสินค้า (ถ้าซื้อ/ขายต่างหน่วยจาก stock UOM)

6

Warehouse (tree) — สร้างคลังปลายทาง (leaf · type=stock)

7

Supplier (supplier-bc) — สำหรับอ้างอิงตอนรับสินค้า/PO

✓ Minimum Data — พร้อมทำ Goods Receipt แรก
  • ✓≥ 1 UOM (ใช้เป็น stock unit)
  • ✓≥ 1 Item Group (leaf)
  • ✓≥ 1 Product (type=single, has_variants=false, มี stock_uom_id + item_group_id)
  • ✓≥ 1 Warehouse (type=stock, is_group=false)
  • ✓≥ 1 Supplier (supplier-bc)
  • +Stock Settings (valuation · default warehouse · negative flag) — ตั้งใน Phase 3
→ inventory รับอะไรไปใช้
  • →product_id + stock_uom_id — สร้าง Lot/movement ในหน่วยที่ถูกต้อง
  • →conversion_factor — แปลงจำนวนซื้อ/ขาย → stock_uom
  • →item_group_id — จัดหมวดในรายงานสต็อก
  • →warehouse_id (leaf) — คลังปลายทางของ Goods Receipt
  • →supplier_id (UUID) — ผูกกับ Lot ที่รับเข้า
GATE · ก่อนเปิด Phase 3

ถ้า master ไม่ครบ (เช่น product ไม่มี stock_uom, warehouse เป็น group) → Goods Receipt จะถูกปฏิเสธที่ระดับ service · การเปลี่ยน master ที่ inventory denormalize ไว้ (ชื่อ/หมวด/ยี่ห้อ) จะยิง event item.updated ไปอัปเดต report-bc

06

Developer Implementation Guide · คำสั่งจาก SA

RULE · i18n FLAT COLUMNS

ฟิลด์ 2 ภาษาเก็บเป็น name_th / name_en (ไม่ใช้ JSONB) · Response ยุบเป็น name: { th, en } ผ่าน LocalizationInterceptor · input (DTO) เป็น flat keys

RULE · BASE ENTITY

ทุกตารางสืบทอด BaseEntity — ได้ id · created_at/by · updated_at/by · is_deleted · deleted_at/by/reason · การลบเป็น soft-delete เสมอ

RULE · NESTED-SET TREE

item_groups และ warehouses เป็น tree (lft/rgt) — rebuild ใน transaction ตอน insert/move · เฉพาะ leaf (is_group=false) ผูกสินค้า/เก็บสต็อกได้

RULE · STOCK UOM IMMUTABLE

stock_uom_id ห้ามแก้ทันทีที่มี stock movement แรก (DB trigger หรือ app guard) กัน available_qty ในอดีตตีความหน่วยผิด

RULE · TEMPLATE NOT TRANSACTABLE

สินค้า has_variants=true ต้องถูกปฏิเสธเมื่อใช้ใน GR/PO/SO ตรง — อนุญาตเฉพาะ variant ที่ generate จาก attribute

RULE · CONVERSION ROUND CHECK

ถ้า stock UOM is_whole_number=true ต้องตรวจว่า qty × conversion_factor เป็นจำนวนเต็ม มิฉะนั้น reject

RULE · UNIQUE KEYS

uq_products_sku · uq_suppliers_code — บังคับ unique ระดับ DB · UOM seed ~200+ รายการเป็น master ตั้งต้น

RULE · UUID CROSS-BC + EVENTS

supplier_id (supplier-bc) และ context_ref (storage-bc) อ้างด้วย UUID ห้าม FK ข้าม DB · การแก้ master ยิง item.updated ให้ report-bc/sales/supplier อัปเดต cache
