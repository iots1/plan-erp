Inventory Core — Lot · FIFO/FEFO · Bundle · Goods Movement

แกนกลางของทั้งระบบ ควบคุมความถูกต้องของสต็อก ต้นทุนต่อล็อต และการหมดอายุ

01

Business Overview & Roles · ใครทำอะไร

WAREHOUSE MANAGER
ผู้จัดการคลังสินค้า

อนุมัติการปรับสต็อก, ตัดจำหน่ายสินค้าหมดอายุ, กำหนดจุดสั่งซื้อ (reorder), เฝ้าระวังล็อตใกล้หมดอายุ

WAREHOUSE STAFF
เจ้าหน้าที่คลัง

บันทึกรับสินค้าจริง (Goods Receipt), ระบุล็อต/ต้นทุน/วันหมดอายุ, หยิบสินค้า (picking) ตามใบจ่าย

QC OFFICER
เจ้าหน้าที่ตรวจรับ

ตรวจคุณภาพก่อนรับเข้าล็อต, ปฏิเสธของชำรุด, ยืนยันจำนวนรับจริงเทียบ PO

วัตถุประสงค์ของเฟส: สต็อกในระบบต้องตรงกับของจริงเสมอ ทุกล็อตต้องมีต้นทุนนำเข้าและวันหมดอายุที่ถูกต้อง และการตัดสต็อกต้องเป็นไปตามกฎ FIFO/FEFO อย่างเคร่งครัด
02

Master Data & Settings · ตั้งค่าสินค้า (Item Master)

ตำแหน่งในระบบ: ทุกหัวข้อในส่วนนี้ — Item, Item Group, Item Attribute, Brand, UOM, UOM Conversion Factor — จัดกลุ่มอยู่ในเมนู "ตั้งค่าสินค้า" (Product / Stock Settings) บน Left Sidebar ของแอปพลิเคชัน เป็น master data ที่ต้องตั้งค่าให้เสร็จก่อนเริ่มบันทึกธุรกรรมใด ๆ (Goods Receipt, PO, SO ฯลฯ)
ITEM GROUP
หมวดหมู่สินค้า

โครงสร้างแบบ ต้นไม้ (tree) ผูก parent/child ได้หลายชั้น (เช่น เครื่องดื่ม → เครื่องดื่มไม่มีแอลกอฮอล์ → น้ำอัดลม) ใช้จัดกลุ่ม, กำหนดค่าเริ่มต้น (คลัง/บัญชี) และกรองรายงานตามหมวด

BRAND
ยี่ห้อสินค้า

รายการยี่ห้อ/ผู้ผลิตแบบเรียบง่าย (ไม่มีลำดับชั้น) ผูกกับ Item เพื่อใช้กรองค้นหาและออกรายงานยอดขายแยกตามยี่ห้อ

UOM
หน่วยนับ (Unit of Measure)

รายการหน่วยกลาง เช่น ชิ้น, กล่อง, กก., ลิตร แต่ละหน่วยระบุ is_whole_number (บังคับจำนวนเต็ม เช่น "ชิ้น" หรืออนุญาตทศนิยม เช่น "กก.")

UOM CONVERSION FACTOR
อัตราแปลงหน่วยต่อสินค้า

กำหนด ต่อรายสินค้า ว่า 1 หน่วยทางเลือก = กี่เท่าของ stock UOM เช่น 1 กล่อง = 12 ชิ้น ใช้แปลงจำนวนตอนซื้อ/ขายเป็นหน่วยที่ต่างจากหน่วยเก็บสต็อก

ITEM ATTRIBUTE
คุณลักษณะสินค้า (สำหรับตัวเลือก/variant)

นิยามแกนตัวเลือก เช่น สี, ขนาด พร้อมรายการค่าที่เป็นไปได้ (attribute values) ใช้ผูกกับ Item ต้นแบบเพื่อ generate สินค้าย่อยแบบ variant (เช่น เสื้อ-แดง-L)

ITEM
สินค้า (ตัวหลัก)

มาสเตอร์สินค้าหลัก อ้างอิง item group, brand, stock UOM และ (ถ้ามี) attribute ต้นแบบสำหรับ variant — เป็นจุดร่วมที่ Lot, PO, SO, Bundle ทั้งหมดผูกกับ

ลำดับการตั้งค่า · Setup Flow

1

ตั้งค่า UOM พื้นฐาน (ชิ้น, กล่อง, กก. ฯลฯ) และระบุว่าหน่วยใด is_whole_number

2

สร้าง Item Group เป็นโครงสร้างต้นไม้ตามที่ธุรกิจต้องการจัดหมวด และสร้าง Brand ตามยี่ห้อที่ขาย

3

(ถ้ามีสินค้าแบบตัวเลือก) สร้าง Item Attribute + attribute values ล่วงหน้า เช่น สี = {แดง, น้ำเงิน}, ขนาด = {S, M, L}

4

สร้าง Item — เลือก item group, brand, stock UOM (หน่วยหลักที่ใช้เก็บสต็อก) และตั้ง has_variants=true พร้อมเลือก attribute หากต้องการให้ระบบ generate สินค้าย่อยอัตโนมัติ

5

ตั้งค่า UOM Conversion Factor ต่อ Item (ถ้าซื้อ/ขายเป็นหน่วยอื่นที่ไม่ใช่ stock UOM) เช่น หน่วยซื้อ = กล่อง (factor 12), หน่วยขาย = ชิ้น (factor 1)

Edge Cases · กรณีขอบเขต

เปลี่ยน Stock UOM ภายหลัง

ถ้า Item มี stock movement เกิดขึ้นแล้ว ห้ามเปลี่ยน stock UOM เพราะจะทำให้ available_qty ในอดีตตีความผิดหน่วย ต้องปิด item เดิมแล้วสร้างใหม่แทน

ลบ Item Group ที่มี Item ผูกอยู่

ห้ามลบ/ต้องย้าย Item ไปกลุ่มอื่นก่อน และห้ามลบ Item Group ที่มี child group อยู่ภายใต้

Conversion Factor กับหน่วยจำนวนเต็ม

ถ้า stock UOM มี is_whole_number=true (เช่น ชิ้น) การแปลงต้องได้ผลลัพธ์เป็นจำนวนเต็มเสมอ ห้ามตั้ง factor ที่ทำให้ปัดเศษ

ใช้ Item ต้นแบบ (template) ทำธุรกรรม

Item ที่ has_variants=true ต้อง บล็อกไม่ให้เลือกใน Goods Receipt/PO/SO โดยตรง — บังคับให้เลือกที่ variant ที่ระบบ generate ไว้เท่านั้น

Stock Settings · ค่าตั้งต้นระบบสต็อก

หน้า Stock Settings (เมนู Settings) เก็บค่า config ระดับระบบที่ business logic ทุกจุดต้องอ่าน — ต้องตั้งให้เสร็จก่อนเปิดใช้งานจริง

valuation_method

กลยุทธ์ตัดสต็อกตั้งต้น: FIFO หรือ FEFO (ต่อ Item override ได้)

allow_negative_stock

อนุญาตให้ตัดจนติดลบหรือไม่ — ค่าเริ่มต้น false (ตรงกับ CHECK ≥ 0)

default_warehouse

คลังตั้งต้นเวลาสร้างเอกสารใหม่ (ลดการกรอกซ้ำ)

expiry_alert_days

จำนวนวันก่อนหมดอายุที่จะยิง expiry.approaching

stock_frozen_upto

ล็อกงวดอดีต: ห้ามบันทึก/แก้ movement ที่ posting_date ≤ วันนี้ (กันแก้ย้อนหลังกระทบบัญชี)

over_receipt_allowance

% ที่ยอมให้รับ/ส่งเกินจำนวนในเอกสารอ้างอิง (เช่น รับเกิน PO ได้ 5%)

03

Core Use Cases & Data Entry · บันทึกข้อมูลยังไง

UC-3.1 · GOODS RECEIPT (รับสินค้าเข้าคลัง)
1

เปิดใบรับสินค้า (GRN) อ้างอิง PO จาก supplier-bc (po_id, supplier_id)

2

กรอกจำนวนรับจริง, ต้นทุน/หน่วย, lot_number, expiry_date, คลังปลายทาง

3

ระบบสร้าง Lot ใหม่ ต่อรายการ กำหนด available_qty = received_qty

4

บันทึก stock_movement (type=RECEIPT) + emit lot.created

UC-3.2 · GOODS ISSUE (ตัดสต็อก FIFO/FEFO)
1

รับคำขอจ่ายออก: product_id, qty, warehouse, strategy

2

เลือกล็อตเรียงตาม received_at (FIFO) หรือ expiry_date (FEFO)

3

ตัด available_qty ทีละล็อต สร้าง allocation (lot_id, qty, unit_cost)

4

บันทึก movement (ISSUE) + emit stock.deducted

UC-3.3 · BUNDLE EXPLOSION (ขายแบบชุด)

เมื่อรายการที่จ่ายเป็น type=bundle ระบบจะ แตกชุด ตาม BOM (bundle_items) × จำนวนชุด → รวม demand ต่อสินค้าย่อย → ตัด FIFO/FEFO ของแต่ละสินค้าย่อย ในทรานแซกชันเดียว. COGS ของชุด = ผลรวมต้นทุนจากทุกล็อตของทุกสินค้าย่อย

UC-3.4 · STOCK ENTRY (รับเข้า / โอนคลัง / เบิกใช้ภายใน / แปรรูป)

เอกสารกลางสำหรับการเคลื่อนไหวสต็อกที่ ไม่ผูกกับ PO/SO — ครอบคลุม 4 ประเภทหลัก:

1

Material Receipt — รับเข้าที่ไม่ผูก PO: ยอดยกมาตอนเปิดระบบ (opening) / ของแถม / ปรับเพิ่ม — สร้างล็อตใหม่พร้อม import_cost + expiry (RECEIPT)

2

Material Transfer — ย้ายล็อตระหว่างคลัง: ตัด available ที่คลังต้นทาง (ISSUE) + สร้าง/เพิ่มล็อตที่คลังปลายทาง (RECEIPT) คงต้นทุนล็อตเดิม ไม่สร้างกำไร/ขาดทุน · รองรับ in-transit warehouse สำหรับการโอนข้ามสาขา

3

Material Issue — เบิกใช้ภายใน (ของเสีย / ตัวอย่าง / ใช้ในองค์กร): ตัด FIFO/FEFO เหมือน Goods Issue แต่ปลายทางเป็นบัญชีค่าใช้จ่าย ไม่ใช่ลูกค้า

4

Repack — แปรรูป: ตัดสินค้าต้นทาง (source items) → สร้างล็อตสินค้าปลายทางใหม่ โดย ม้วนต้นทุนต้นทางเข้าเป็น import_cost ของล็อตใหม่ (value in = value out)

5

ทุกประเภทบันทึก stock_movement คู่ (out/in) ใน ทรานแซกชันเดียว + emit stock.adjusted / lot.created

Core Operational Reports · รายงานขั้นต่ำ (query ตรงจาก inventory-bc)

STOCK LEDGER · บัญชีแยกประเภทสต็อก

ไล่ทุก stock_movement ตามเวลา (in/out/adjust) พร้อม running balance ต่อ item×warehouse×lot — เป็น source of truth สำหรับตรวจสอบ ต่างจากรายงานเชิงวิเคราะห์ที่อยู่ Phase 6

STOCK BALANCE · ยอดคงเหลือ

สรุป available_qty คงเหลือ ณ ปัจจุบัน แยกตามคลัง / ล็อต / วันหมดอายุ + มูลค่าคงคลัง (Σ qty×import_cost) ใช้ตรวจกับของจริง

* รายงานทั้ง 2 นี้ query จาก transactional DB ของ inventory-bc โดยตรง (operational) — ส่วนรายงานเชิงวิเคราะห์/แดชบอร์ด (ageing, trends, analytics) ไปทำใน report-bc (Phase 6 · CQRS)

Edge Cases · กรณีขอบเขต

สต็อกไม่พอ

ตอน Goods Issue หากรวมทุกล็อตแล้วไม่พอ → throw InsufficientStock และ rollback ทั้งรายการ (all-or-nothing) ห้ามตัดบางส่วน

โอนคลังต้องคงต้นทุน

Material Transfer ห้ามเปลี่ยน import_cost ของล็อต — ปลายทางต้องได้ต้นทุนเท่าต้นทาง (ไม่ใช่การซื้อใหม่)

ล็อตหมดอายุ

FIFO/FEFO ต้อง ข้ามล็อตที่ expiry < วันนี้ ไม่นำไปขาย และแจ้งเตือนให้ทำ write-off ผ่านใบปรับสต็อกแยก

รับไม่ครบ PO

Partial receipt → สร้างล็อตเท่าที่รับจริง PO ยังคงเปิดค้าง (สถานะ PARTIAL) รอรับส่วนที่เหลือ

Bundle ขาดชิ้น

หากสินค้าย่อยตัวใดในชุดสต็อกไม่พอ → rollback ทั้งชุด ไม่ตัดสินค้าย่อยตัวอื่นทิ้งไว้

Developer Implementation Guide · คำสั่งจาก SA

RULE · TRANSACTION

การตัดสต็อกทั้งหมด (รวมทุก component ของ bundle) ต้องอยู่ใน Database Transaction เดียว — ถ้าล้มเหลวส่วนใด rollback ทั้งหมด

RULE · CONCURRENCY LOCK

ก่อนอ่าน available_qty ต้องล็อกแถวล็อตด้วย pessimistic write lock (SELECT ... FOR UPDATE) เพื่อกัน race condition ระหว่างการตัด FIFO พร้อมกันหลาย request

RULE · ORDERING

FEFO: ORDER BY expiry_date ASC, received_at ASC · FIFO: ORDER BY received_at ASC — ต้อง tie-break ให้ deterministic

RULE · COST SNAPSHOT

unit_cost ในทุก movement ต้องเป็น snapshot ต้นทุนจริงของล็อต (immutable) ห้ามคำนวณย้อนหลังจากราคาเฉลี่ย

RULE · EXPIRY FILTER

การตัดปกติต้อง WHERE expiry_date > now() — ห้ามขายล็อตหมดอายุ

RULE · STOCK UOM IMMUTABLE

stock_uom_id ของ Item ห้ามแก้ไขทันทีที่มี stock_movement แรกเกิดขึ้นแล้ว (บังคับด้วย DB trigger หรือ application guard) กัน available_qty ในอดีตตีความหน่วยผิด

RULE · TEMPLATE ITEM NOT TRANSACTABLE

Item ที่ has_variants=true ต้องถูกปฏิเสธที่ระดับ service เมื่อถูกใช้ใน Goods Receipt/Issue/PO/SO โดยตรง — อนุญาตเฉพาะ variant ที่ generate จาก Item Attribute เท่านั้น

RULE · CONVERSION ROUND CHECK

ก่อนบันทึกจำนวนที่แปลงหน่วยแล้ว หาก stock UOM ปลายทางมี is_whole_number=true ต้องตรวจว่าไม่มีเศษ (qty × conversion_factor เป็นจำนวนเต็ม) มิฉะนั้น reject

RULE · LEDGER SINGLE SOURCE

Stock Ledger / Balance ต้อง derive จาก stock_movements เท่านั้น (append-only, immutable) ห้ามมี "ยอดคงเหลือ" เก็บซ้ำที่แก้มือได้ · available_qty บนล็อตเป็น cache ที่ต้อง reconcile กับผลรวม movement ได้เสมอ

RULE · STOCK ENTRY ATOMIC PAIR

Stock Entry แบบ TRANSFER/REPACK ต้องเขียน movement คู่ out+in ในทรานแซกชันเดียว — ยอดรวมของระบบต้องไม่เปลี่ยน (conservation) ยกเว้น ISSUE ที่ตัดออกจริง

RULE · SETTINGS DRIVEN

business logic (negative stock, default strategy, expiry days) ต้อง อ่านจาก Stock Settings ไม่ hardcode · ต่อ Item ที่ override ได้ให้ item ชนะ setting ระดับระบบ

RULE · NESTED-SET TREE

Item Group และ Warehouse เป็น tree — ใช้ nested set (lft/rgt) หรือ recursive CTE เพื่อ query ทั้ง subtree ได้เร็ว (รายงานตามหมวด/สาขา) · การ insert/move node ต้อง rebuild lft/rgt ใน transaction · เฉพาะ leaf (is_group=false) เท่านั้นที่ผูก Item/เก็บสต็อกได้

RULE · RUNNING BALANCE + CACHE

แต่ละ movement เก็บ qty_after_transaction (running balance) ทำให้ Stock Ledger อ่านตรง ๆ · แนะนำตาราง cache balance ต่อ item×warehouse (actual/reserved/valuation) เพื่อให้ Stock Balance เร็ว โดย cache ต้อง derive ได้จาก ledger เสมอ

RULE · ABSOLUTE RECONCILIATION

Stock Reconciliation รับ จำนวนที่นับได้จริง (absolute) แล้วให้ระบบคำนวณ qty_delta = counted − current เอง แล้วสร้าง adjustment movement — ไม่ให้ผู้ใช้กรอก delta ตรง ๆ (กันคำนวณผิด)

RULE · SUBMITTABLE DOCUMENTS

เอกสารเคลื่อนไหวสต็อกทุกใบ (GRN, Delivery, Stock Entry, Reconciliation) เป็น submittable: DRAFT → SUBMITTED → CANCELLED · สต็อกจะเคลื่อน (post ledger) เฉพาะตอน submit เท่านั้น · การ cancel ต้อง reverse movement ที่โพสต์ไว้ ไม่ใช่ลบทิ้ง · draft แก้ได้อิสระโดยไม่กระทบสต็อก

RULE · OUTBOX / IDEMPOTENCY

ส่ง event หลัง commit ด้วย Outbox pattern และ consumer ต้อง idempotent (เก็บ processed event id) กัน event ซ้ำจาก RabbitMQ

RULE · UUID ONLY

อ้างอิงข้าม BC ด้วย UUID เท่านั้น · ห้าม Foreign Key ข้ามฐานข้อมูล · available_qty ใช้ CHECK constraint ≥ 0

B.5 · Transaction boundary = submit (concurrency + locking)
await this.dataSource.transaction('READ COMMITTED', async (mgr) => {
  const grn = await mgr.findOneOrFail(GoodsReceipt, {
    where: { id },
    lock: { mode: 'pessimistic_write' },   // ล็อก header กัน submit ซ้ำ (idempotent)
  });
  if (grn.status !== 'DRAFT') throw new ConflictException('goods_receipt.not_draft');
  assertNotFrozen(grn.posting_date, settings.stock_frozen_upto);
  await this.assertWithinPoAllowance(grn);   // TCP → supplier-bc.getPurchaseOrder

  for (const line of grn.items) {
    const qty = convertToStockUom(line);        // integer-safe ถ้า is_whole_number
    const lot = await mgr.save(mgr.create(Lot, {
      product_id: line.product_id, warehouse_id: line.warehouse_id,
      lot_number: line.lot_number, import_cost: line.import_cost,
      received_qty: qty, available_qty: qty,      // CHECK (available_qty >= 0)
      expiry_date: line.expiry_date, supplier_id: grn.supplier_id,
      received_at: grn.posting_date,
    }));
    await mgr.save(mgr.create(StockMovement, {
      lot_id: lot.id, type: 'RECEIPT', qty,
      unit_cost: line.import_cost, source_doc_id: grn.id,   // snapshot ต้นทุนจริง
    }));
    await this.bumpBalanceCache(mgr, line, +qty);   // SELECT ... FOR UPDATE ก่อน +
    line.lot_id = lot.id;
  }
  grn.status = 'SUBMITTED';
  await mgr.save(grn);
  await this.outbox.stage(mgr, events);   // Outbox: relay ส่ง event หลัง commit
});
LOCK · HEADER

ล็อก GRN header ด้วย pessimistic_write ตอน submit → 2 request submit พร้อมกันจะไม่ double-post (idempotent)

LOCK · BALANCE CACHE

ก่อน increment balance cache (Bin) ต่อ item×warehouse ต้อง SELECT ... FOR UPDATE — แถวยอดรวมคือ hot row ที่ต้องกัน lost update (วินัยเดียวกับ FIFO engine)

OUTBOX · AFTER COMMIT

stage lot.created / goods_receipt.submitted ในทรานแซกชันเดียวกับ ledger → relay publish เข้า RabbitMQ หลัง commit · consumer ต้อง idempotent (เก็บ processed event id)

B.7 · Edge Cases & Validations (เฉพาะ Goods Receipt)
Variant template

product.has_variants=true → reject product.is_template_cannot_transact ต้องเลือก variant ที่ generate ไว้

Over-receipt vs PO

(received + รับไปแล้ว) เทียบ ordered × (1 + over_receipt_allowance) · เกิน → 422 over_receipt.exceeded

Frozen period

posting_date ≤ stock_frozen_upto → 422 posting_date.frozen

Negative / zero

received_qty ≥ 1 · import_cost ≥ 0 · DB CHECK available_qty ≥ 0

UOM rounding

ถ้า stock UOM is_whole_number แล้วแปลงได้เศษ → 422 uom.non_integer_conversion

Cancel หลังถูกตัด

Group warehouse / expiry ในอดีต

// ตอน submit ส่งกลับเป็น FLAT keys ตรง DTO (remark_th/remark_en + UUID) เสมอ

ใช้ React Hook Form + Zod schema สะท้อน DTO ฝั่ง backend · Axios interceptor แนบ Accept-Language · Nested object ใช้เฉพาะฝั่ง response — ตอนส่งกลับ flat เสมอ

C.3 · Eventual Consistency
SYNC · งานสต็อกยืนยันทันที

lots + movements commit ในทรานแซกชันเดียวก่อน HTTP 200 → เจ้าหน้าที่เห็นเลขล็อตที่สร้างทันทีจาก response (authoritative) แสดง GRN เป็น SUBMITTED ได้เลย

ASYNC · fan-out ตามหลัง

โชว์ toast รอง "กำลัง sync ไปรายงาน & PO…" สำหรับ consumer async (report-bc, supplier-bc) — อย่า block UI · สถานะ PO (PARTIAL/COMPLETED) ที่ supplier-bc เป็นเจ้าของอาจ lag ไม่กี่วินาที → หน้า PO ให้ refetch/poll อย่าเดาว่าพลิกทันที
