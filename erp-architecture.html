01 — Executive Summary

บทสรุปผู้บริหาร

ระบบเดิมเป็น Legacy ERP System ที่รวมงาน คลังสินค้า (Stock), หน้าร้าน (Store/POS) และ บัญชี (Accounting) ไว้ในฐานข้อมูลก้อนเดียว (Monolith) ทำให้ขยายยาก แก้ไขเสี่ยง และไม่สามารถคำนวณต้นทุน–กำไรระดับ Lot ได้แม่นยำ

เป้าหมายคือการออกแบบใหม่เป็น Microservices แบบ DDD บน Monorepo (Nx + pnpm) โดยแต่ละ Bounded Context เป็นเจ้าของฐานข้อมูลของตนเอง ห้าม Join ข้ามฐานข้อมูล และอ้างอิงข้อมูลข้ามบริบทด้วย UUID เท่านั้น เพื่อให้แต่ละบริการ Deploy และ Scale ได้อิสระ

หัวใจของโครงการคือความถูกต้องของ การตัดสต็อกแบบ FIFO/FEFO, การจัดการ Lot & วันหมดอายุ และ การคำนวณ COGS และกำไร/ขาดทุนต่อ Lot ซึ่งอิงต้นทุนนำเข้าจริงของแต่ละล็อต ไม่ใช่ต้นทุนเฉลี่ย

Target Outcomes
  • →8 บริการอิสระ แยกฐานข้อมูลตามบริบทธุรกิจ
  • →ต้นทุนแม่นยำ ระดับ Lot ทุกธุรกรรม
  • →2 ภาษา (ไทย/อังกฤษ) ครบทั้ง UI, ข้อความระบบ และ Master Data
  • →Event-Driven ผ่าน RabbitMQ ลดการผูกกันระหว่างบริการ
  • →ไฟล์รวมศูนย์ บน MinIO พร้อม Presigned URL
02 — Architecture Blueprint

พิมพ์เขียวสถาปัตยกรรม

ทุกบริการเขียนด้วย NestJS 11 บน Fastify อยู่ในเวิร์กสเปซเดียวกัน (Nx) Frontend เป็น Next.js เรียกใช้ผ่าน API Gateway การสื่อสารระหว่างบริการแบ่งเป็น 2 รูปแบบชัดเจน: RabbitMQ สำหรับเหตุการณ์แบบ async (pub/sub) และ TCP สำหรับการร้องขอแบบ sync (request/response)

System Context Map · Database-per-Context

TCP — Synchronous

ใช้เมื่อผู้เรียก “ต้องรอคำตอบทันที” เช่น Gateway ถามสิทธิ์จาก iam, ตรวจ Availability จาก inventory-bc หรือขอ Presigned URL จาก storage-bc. ใช้ NestJS Microservice Transport แบบ TCP

RabbitMQ — Asynchronous

ใช้เผยแพร่ Domain Event แบบ fire-and-forget เช่น delivery.confirmed, stock.deducted, lot.created. ผู้สนใจ (เช่น finance-bc, report-bc) Subscribe เองโดยผู้ส่งไม่ต้องรับรู้ ทำให้ระบบ Decoupled

03 — Internationalization

กลยุทธ์ 2 ภาษา (ไทย / อังกฤษ)

ระบบต้องรองรับ ไทยและอังกฤษ ครบทุกชั้น แบ่งความรับผิดชอบชัดเจนระหว่าง Frontend, Backend และ Database

LAYER 01

Next.js — UI & Header

จัดการ Routing ตามภาษา, แปลข้อความ UI ด้วยไลบรารี i18n และแนบ Header Accept-Language: th ในทุก Request ผ่าน Axios Interceptor

NestJS — nestjs-i18n

PostgreSQL — Flat Columns

ทุกฟิลด์ 2 ภาษาแยกเป็นคอลัมน์คู่ขนานเสมอ name_th / name_en เพื่อประสิทธิภาพสูงสุดในการทำ Index และ Full-Text Search ไม่ใช้ JSONB สำหรับฟิลด์ทั่วไป

05 — Critical Business Logic

Deep Dive: FIFO · Lot · กำไร/ขาดทุน

นี่คือกฎธุรกิจที่ห้ามผิดพลาด inventory-bc และ finance-bc ทำงานร่วมกันเพื่อรับประกันว่าต้นทุนที่ตัดออก และกำไรที่คำนวณ อิงต้นทุนนำเข้าจริงของ แต่ละ Lot เสมอ

RULE 01

Lot & Expiry

สินค้ารับเข้าทุกครั้งต้องผูก Lot ที่บันทึก ต้นทุนนำเข้า และ วันหมดอายุ เฉพาะของล็อตนั้น

RULE 02

FIFO / FEFO

ตัดสต็อกจากล็อตที่ เข้าก่อน (FIFO) หรือ หมดอายุก่อน (FEFO) ก่อนเสมอ โดยอัตโนมัติ

RULE 03

P&L ต่อ Lot

COGS และกำไร/ขาดทุน คำนวณจากต้นทุนจริงของล็อตที่ถูกตัด ไม่ใช่ต้นทุนเฉลี่ย

Single vs Bundle · การขายแบบเดี่ยว & แบบชุด

ขายแบบชุด (Bundle) ต้องตัด Lot ให้ถูกต้อง

สินค้ามี 2 ชนิด: เดี่ยว (single) ตัดล็อตของตัวเองตรง ๆ และ ชุด (bundle/kit) ที่ประกอบจากสินค้าย่อยหลายรายการตาม BOM เมื่อขายชุด ระบบต้อง “แตกชุด” (explode) เป็นสินค้าย่อยก่อน แล้วตัด FIFO/FEFO ของ แต่ละสินค้าย่อยแยกกัน — COGS ของชุด = ผลรวมต้นทุนจากทุกล็อตของทุกสินค้าย่อย

ตัวอย่าง · ขาย “ชุดของขวัญ A” จำนวน 10 ชุด

X-LOT25 @ ฿32 = ฿160
COGS X฿610
ต้องตัด · สินค้า Y = 10 หน่วย
Y-LOT110 @ ฿48 = ฿480

COGS ชุดของขวัญ A (10 ชุด) = ฿610 + ฿480 = ฿1,090 — แต่ละสินค้าย่อยตัดตามลำดับ FEFO ของตัวเอง และทั้งหมดต้องอยู่ใน ทรานแซกชันเดียว (all-or-nothing) หากสินค้าย่อยตัวใดสต็อกไม่พอ ต้อง rollback ทั้งชุด

รายละเอียดต่อไปนี้ถูก ยืนยัน/เพิ่มเข้า scope หลังตรวจ schema (736 ตาราง) และ controller logic ของระบบสต็อกเชิงอ้างอิง ทำให้ประมาณการงานสมจริงขึ้น (รวม ~+40–60 ชม. กระจายใน P0/P1):

  • 01Item Group + Warehouse เป็น tree (nested-set lft/rgt) ไม่ใช่ parent FK เฉย ๆ — ต้องเขียน logic rebuild lft/rgt ตอน insert/move (BE +16)
  • 02Ledger เก็บ running balance ต่อ movement + ตาราง cache balance ต่อ item×warehouse (แบบ Bin) เพื่อให้ Stock Balance เร็ว (BE +16)
  • 03Stock Entry เพิ่มเป็น 4 ประเภท (Receipt / Transfer / Issue / Repack) + รองรับ in-transit warehouse ตอนโอนข้ามสาขา (BE +8)
  • 04Stock Reconciliation ใช้จำนวน absolute (นับได้จริง) แล้วระบบคำนวณ delta เอง — เปลี่ยนจากกรอก qty_delta ตรง ๆ
  • 05Stock Settings เพิ่ม frozen period + over-receipt allowance · เตรียม seed หน่วยนับมาตรฐาน (UOM) ~200+ รายการเป็น master data ตั้งต้น (BE +8)
  • 06ทุกเอกสารธุรกรรมเป็น submittable (draft → submit → cancel) — ต้องมี state machine + logic โพสต์/ยกเลิก ledger ทุกใบ (GR, Delivery, Stock Entry, Reconciliation) ไม่ใช่ CRUD ธรรมดา
  • 07งานที่หนักสุดคือ valuation engine (FIFO/FEFO ต่อ warehouse×batch + recompute) — ในระบบอ้างอิงเป็นโค้ด ~2,600 บรรทัด · และ ฟอร์ม Item ใหญ่มาก (79 ฟิลด์ + 8 child tables: barcode, reorder, UOM conversion, variant, default, supplier ฯลฯ) → FE ต้องทำแบบ tabbed form
Phase 4 · Procurement & Sales — ที่พบจริง (+72 ชม.)
  • 01Pricing Rule engine — ราคาพิเศษ/ส่วนลดตามจำนวน·ช่วงวันที่·ลูกค้า เป็นเครื่องมือแยก ไม่ใช่ field ธรรมดา (BE +32)

Phase 5 · Finance — ที่พบจริง (+88 ชม.)
  • 01GL = double-entry ledger (debit=credit ต่อ account, immutable, post/cancel) — เป็น engine คู่ขนานกับ stock ledger, งานหลักของเฟส (BE +32)
  • 02Payment Entry + allocation — จ่าย/รับเงินแล้ว allocate ไปหลายใบกำกับ (AR/AP) พร้อม over/under-payment (BE +32)

ใช้รูปแบบ Strangler Fig — รันระบบใหม่คู่ขนานกับระบบเดิม แล้วค่อย ๆ ย้ายทีละโมดูล ลดความเสี่ยงและไม่ต้องหยุดธุรกิจ

การย้ายข้อมูล (ETL)

  • 01สร้างตาราง legacy_id_map จับคู่ ID เดิม ↔ UUID ใหม่ ทุกบริการ
  • 02แปลง field เดิมเข้าสู่ schema ใหม่: ชื่อสินค้า → name.th, ประเภท → category, ราคาต้นทุน/ราคาขาย/ภาษี
  • 03ยอดคงเหลือปัจจุบัน (จำนวน) แปลงเป็น “Migration Lot” ตั้งต้น 1 ล็อต/สินค้า พร้อมต้นทุนล่าสุดและวันหมดอายุ
  • 04แยกข้อมูลลูกค้า/ผู้จัดซื้อไปยัง sales-bc / supplier-bc ตามบริบท

การย้ายไฟล์สู่ MinIO

  • 01Bulk copy โฟลเดอร์รูปสินค้า/เอกสารเดิมเข้าสู่ MinIO buckets ด้วย mc mirror หรือ SDK
  • 02ลงทะเบียน file_objects พร้อม context_ref = UUID ของสินค้า/เอกสารที่แมปไว้
  • 03ตรวจ Checksum และจำนวนไฟล์หลังย้าย ก่อนตัดการเข้าถึงพาธเดิม
  • 04Frontend เปลี่ยนมาเรียก Presigned URL จาก storage-bc แทน static path เดิม
