PHASE 3 inventory-bc · Critical Path

Inventory Core — Lot · FIFO/FEFO · Bundle · Goods Movement

แกนกลางของทั้งระบบ ควบคุมความถูกต้องของสต็อก ต้นทุนต่อล็อต และการหมดอายุ

01

Business Overview & Roles · ใครทำอะไร

WAREHOUSE MANAGER
ผู้จัดการคลังสินค้า

อนุมัติการปรับสต็อก, ตัดจำหน่ายสินค้าหมดอายุ, กำหนดจุดสั่งซื้อ (reorder), เฝ้าระวังล็อตใกล้หมดอายุ

WAREHOUSE STAFF
เจ้าหน้าที่คลัง

บันทึกรับสินค้าจริง (Goods Receipt), ระบุล็อต/ต้นทุน/วันหมดอายุ, หยิบสินค้า (picking) ตามใบจ่าย

QC OFFICER
เจ้าหน้าที่ตรวจรับ

ตรวจคุณภาพก่อนรับเข้าล็อต, ปฏิเสธของชำรุด, ยืนยันจำนวนรับจริงเทียบ PO

วัตถุประสงค์ของเฟส: สต็อกในระบบต้องตรงกับของจริงเสมอ ทุกล็อตต้องมีต้นทุนนำเข้าและวันหมดอายุที่ถูกต้อง และการตัดสต็อกต้องเป็นไปตามกฎ FIFO/FEFO อย่างเคร่งครัด

Core Use Cases & Data Entry · บันทึกข้อมูลยังไง

UC-3.1 · GOODS RECEIPT (รับสินค้าเข้าคลัง)
1

เปิดใบรับสินค้า (GRN) อ้างอิง PO จาก supplier-bc (poId, supplierId)

2

กรอกจำนวนรับจริง, ต้นทุน/หน่วย, lot_number, expiry_date, คลังปลายทาง

3

ระบบสร้าง Lot ใหม่ ต่อรายการ กำหนด available_qty = received_qty

4

บันทึก stock_movement (type=RECEIPT) + emit lot.created

UC-3.2 · GOODS ISSUE (ตัดสต็อก FIFO/FEFO)
1

รับคำขอจ่ายออก: productId, qty, warehouse, strategy

2

เลือกล็อตเรียงตาม receivedAt (FIFO) หรือ expiryDate (FEFO)

3

ตัด available_qty ทีละล็อต สร้าง allocation (lotId, qty, unitCost)

4

บันทึก movement (ISSUE) + emit stock.deducted

UC-3.3 · BUNDLE EXPLOSION (ขายแบบชุด)

เมื่อรายการที่จ่ายเป็น type=bundle ระบบจะ แตกชุด ตาม BOM (bundle_items) × จำนวนชุด → รวม demand ต่อสินค้าย่อย → ตัด FIFO/FEFO ของแต่ละสินค้าย่อย ในทรานแซกชันเดียว. COGS ของชุด = ผลรวมต้นทุนจากทุกล็อตของทุกสินค้าย่อย

Edge Cases · กรณีขอบเขต

สต็อกไม่พอ

ตอน Goods Issue หากรวมทุกล็อตแล้วไม่พอ → throw InsufficientStock และ rollback ทั้งรายการ (all-or-nothing) ห้ามตัดบางส่วน

ล็อตหมดอายุ

FIFO/FEFO ต้อง ข้ามล็อตที่ expiry < วันนี้ ไม่นำไปขาย และแจ้งเตือนให้ทำ write-off ผ่านใบปรับสต็อกแยก

รับไม่ครบ PO

Partial receipt → สร้างล็อตเท่าที่รับจริง PO ยังคงเปิดค้าง (สถานะ PARTIAL) รอรับส่วนที่เหลือ

Bundle ขาดชิ้น

หากสินค้าย่อยตัวใดในชุดสต็อกไม่พอ → rollback ทั้งชุด ไม่ตัดสินค้าย่อยตัวอื่นทิ้งไว้

Developer Implementation Guide · คำสั่งจาก SA

RULE · TRANSACTION

การตัดสต็อกทั้งหมด (รวมทุก component ของ bundle) ต้องอยู่ใน Database Transaction เดียว — ถ้าล้มเหลวส่วนใด rollback ทั้งหมด

RULE · CONCURRENCY LOCK

ก่อนอ่าน available_qty ต้องล็อกแถวล็อตด้วย pessimistic write lock (SELECT ... FOR UPDATE) เพื่อกัน race condition ระหว่างการตัด FIFO พร้อมกันหลาย request

RULE · ORDERING

FEFO: ORDER BY expiry_date ASC, received_at ASC · FIFO: ORDER BY received_at ASC — ต้อง tie-break ให้ deterministic

RULE · COST SNAPSHOT

unit_cost ในทุก movement ต้องเป็น snapshot ต้นทุนจริงของล็อต (immutable) ห้ามคำนวณย้อนหลังจากราคาเฉลี่ย

RULE · EXPIRY FILTER

การตัดปกติต้อง WHERE expiry_date > now() — ห้ามขายล็อตหมดอายุ

RULE · OUTBOX / IDEMPOTENCY

ส่ง event หลัง commit ด้วย Outbox pattern และ consumer ต้อง idempotent (เก็บ processed event id) กัน event ซ้ำจาก RabbitMQ

RULE · UUID ONLY

อ้างอิงข้าม BC ด้วย UUID เท่านั้น · ห้าม Foreign Key ข้ามฐานข้อมูล · available_qty ใช้ CHECK constraint ≥ 0

Core Use Cases & Data Entry · บันทึกข้อมูลยังไง

กระบวนการขาย · Quotation → SO → Delivery

1

สร้างใบเสนอราคา (Quotation) — เลือกลูกค้า, สินค้า, unit price, discount, ภาษี, วันหมดอายุใบเสนอราคา (valid_until) → สถานะ DRAFT/SENT

2

ลูกค้าอนุมัติ → Quotation เปลี่ยนสถานะเป็น APPROVED

3

แปลงเป็น Sales Order (SO) — คัดลอกรายการ + ตรวจ availability จาก inventory ผ่าน TCP (sync) → สถานะ CONFIRMED (ยังไม่ตัดสต็อก)

4

ออกใบส่งของ (Delivery Note) → emit delivery.confirmed → inventory ตัดสต็อก FIFO/FEFO และส่ง allocations กลับมาเก็บใน delivery item

กระบวนการจัดซื้อ · Purchase Order

1

สร้าง PO — เลือก supplier, สินค้า (productId ref inventory), จำนวน, ต้นทุนที่ตกลง, วันส่งมอบ → DRAFT

2

อนุมัติ & ส่ง → APPROVED → SENT (emit po.approved)

3

รับสินค้า (เชื่อม UC-3.1) → emit po.received → inventory สร้างล็อต → PO เป็น PARTIAL/RECEIVED

Edge Cases · กรณีขอบเขต

สต็อกไม่พอตอน SO

ตอน confirm SO ตรวจแล้วสต็อกไม่พอ → เตือน และตาม policy: อนุญาต backorder หรือ block. ไม่ตัดสต็อกจนกว่าจะออกใบส่งของ

Quotation หมดอายุ

หาก valid_until ผ่านไปแล้ว ห้ามแปลงเป็น SO ต้องต่ออายุหรือออกใบใหม่

ราคาพิเศษ

รองรับราคาพิเศษ 1/2/3 ตามช่วงวันที่ (เริ่ม–ถึง) — ระบบเลือกราคาที่มีผลอัตโนมัติ

รับของไม่ครบ PO

รับได้บางส่วน → PO สถานะ PARTIAL รอรับส่วนที่เหลือ ไม่ปิด PO อัตโนมัติ

Developer Implementation Guide · คำสั่งจาก SA

RULE · NO CROSS-DB QUERY

sales-bc / supplier-bc ห้าม query ตาราง products ของ inventory ตรง ๆ ให้เรียกผ่าน TCP หรือใช้ replicated read model

RULE · SYNC CHECK, ASYNC DEDUCT

ตรวจ availability ตอน confirm SO เป็น TCP sync แต่การตัดสต็อกจริงเป็น event async — ยอมรับ eventual consistency

RULE · IDEMPOTENT DELIVERY

การยิง delivery.confirmed ต้อง idempotent (มี delivery id) ป้องกันการตัดสต็อกซ้ำเมื่อ event ถูกส่งซ้ำ

RULE · DENORMALIZE SNAPSHOT

เก็บ ชื่อสินค้า + ราคา ณ เวลาออกเอกสาร ในตาราง item เพราะ master data อาจเปลี่ยนภายหลัง

RULE · STATE MACHINE

สถานะเอกสารเป็น state machine ชัดเจน (DRAFT→…→CLOSED) ห้าม skip state และตรวจ transition ที่ถูกต้อง

RULE · SAGA + COMPENSATION

ใช้ Saga สำหรับ SO→delivery→stock→invoice หากตัดสต็อกล้มเหลว ต้อง compensate (delivery กลับเป็น PENDING)

RULE · MONEY TYPE

จำนวนเงินใช้ decimal หรือ integer (สตางค์) ห้ามใช้ float

Developer Implementation Guide · คำสั่งจาก SA

RULE · EXACT LOT COST

COGS = Σ(allocation.qty × allocation.unit_cost) ต้องใช้ unit_cost ที่มากับ event เท่านั้น ห้ามคำนวณเฉลี่ย และห้าม query ราคาปัจจุบันของสินค้า

RULE · PER-ALLOCATION P&L

คำนวณกำไรต่อ allocation แล้วค่อย aggregate — หนึ่ง delivery อาจกินหลายล็อตที่ต้นทุนต่างกัน

RULE · IMMUTABLE COGS

COGS เก็บเป็น snapshot immutable แก้ไขผ่าน adjusting entry เท่านั้น

RULE · GAPLESS NUMBERING

เลขที่ใบกำกับต้อง gapless & transaction-safe (ใช้ advisory lock / sequence table) กันเลขซ้ำ/ข้าม

RULE · DECIMAL & ROUNDING

ใช้ decimal(18,4) หรือ integer สตางค์ · ปัดเศษตามกฎภาษี · ห้าม float

RULE · IDEMPOTENT CONSUMER

consumer ของ stock.deducted ต้อง idempotent (processed_events) กันการสร้าง COGS ซ้ำ

RULE · INVOICE IMMUTABILITY

ใบกำกับที่ออกแล้ว immutable → แก้ไขด้วย credit/debit note pattern เท่านั้น

Developer Implementation Guide · คำสั่งจาก SA

RULE · DENORMALIZE FOR READ

Read model ออกแบบเพื่อ อ่านเร็ว — ยอมให้ข้อมูลซ้ำ/denormalized เก็บชื่อสินค้า (JSONB) มาด้วย ไม่ต้อง join ข้ามบริการตอน query

RULE · IDEMPOTENT PROJECTION

ทุก consumer ต้อง idempotent ผ่านตาราง processed_events (event_id unique) กันการนับซ้ำ

RULE · HANDLE OUT-OF-ORDER

ใช้ version / updated_at + UPSERT เพื่อกัน event เก่าทับใหม่

RULE · NO SYNC CALL ON QUERY

ตอน query dashboard ห้ามเรียก transactional service — ต้อง pre-compute ล่วงหน้าเท่านั้น

RULE · SEPARATE READ DB

report-bc มีฐานข้อมูลของตัวเอง แยกจากบริการอื่น สามารถ rebuild ได้จาก replay

RULE · EVENT-DRIVEN REFRESH

อัปเดต summary ตาม event ที่เข้ามา ไม่ใช้ polling · ใช้ index ครบเพื่อให้ query < 200ms

RULE · SHOW FRESHNESS

แสดง last_updated บน dashboard เพื่อสื่อสาร eventual consistency ให้ผู้ใช้ทราบ
