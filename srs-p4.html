02

Core Use Cases & Data Entry · บันทึกข้อมูลยังไง

กระบวนการขาย · Quotation → SO → Delivery

1

สร้างใบเสนอราคา (Quotation) — เลือกลูกค้า, สินค้า, unit price, discount, ภาษี, วันหมดอายุใบเสนอราคา (valid_until) → สถานะ DRAFT/SENT

2

ลูกค้าอนุมัติ → Quotation เปลี่ยนสถานะเป็น APPROVED

3

แปลงเป็น Sales Order (SO) — คัดลอกรายการ + ตรวจ availability จาก inventory ผ่าน TCP (sync) → สถานะ CONFIRMED (ยังไม่ตัดสต็อก)

4

ออกใบส่งของ (Delivery Note) → emit delivery.confirmed → inventory ตัดสต็อก FIFO/FEFO และส่ง allocations กลับมาเก็บใน delivery item

กระบวนการจัดซื้อ · Purchase Order

1

สร้าง PO — เลือก supplier, สินค้า (product_id ref inventory), จำนวน, ต้นทุนที่ตกลง, วันส่งมอบ → DRAFT

2

อนุมัติ & ส่ง → APPROVED → SENT (emit po.approved)

3

รับสินค้า (เชื่อม UC-3.1) → emit po.received → inventory สร้างล็อต → PO เป็น PARTIAL/RECEIVED

ข้อมูลที่บันทึกในแต่ละขั้น

Edge Cases · กรณีขอบเขต

สต็อกไม่พอตอน SO

ตอน confirm SO ตรวจแล้วสต็อกไม่พอ → เตือน และตาม policy: อนุญาต backorder หรือ block. ไม่ตัดสต็อกจนกว่าจะออกใบส่งของ

Quotation หมดอายุ

หาก valid_until ผ่านไปแล้ว ห้ามแปลงเป็น SO ต้องต่ออายุหรือออกใบใหม่

ราคาพิเศษ

รองรับราคาพิเศษ 1/2/3 ตามช่วงวันที่ (เริ่ม–ถึง) — ระบบเลือกราคาที่มีผลอัตโนมัติ

รับของไม่ครบ PO

รับได้บางส่วน → PO สถานะ PARTIAL รอรับส่วนที่เหลือ ไม่ปิด PO อัตโนมัติ

Developer Implementation Guide · คำสั่งจาก SA

RULE · NO CROSS-DB QUERY

sales-bc / supplier-bc ห้าม query ตาราง products ของ inventory ตรง ๆ ให้เรียกผ่าน TCP หรือใช้ replicated read model

RULE · SYNC CHECK, ASYNC DEDUCT

ตรวจ availability ตอน confirm SO เป็น TCP sync แต่การตัดสต็อกจริงเป็น event async — ยอมรับ eventual consistency

RULE · IDEMPOTENT DELIVERY

การยิง delivery.confirmed ต้อง idempotent (มี delivery id) ป้องกันการตัดสต็อกซ้ำเมื่อ event ถูกส่งซ้ำ

RULE · DENORMALIZE SNAPSHOT

เก็บ ชื่อสินค้า + ราคา ณ เวลาออกเอกสาร ในตาราง item เพราะ master data อาจเปลี่ยนภายหลัง

RULE · STATE MACHINE

สถานะเอกสารเป็น state machine ชัดเจน (DRAFT→…→CLOSED) ห้าม skip state และตรวจ transition ที่ถูกต้อง

RULE · SAGA + COMPENSATION

ใช้ Saga สำหรับ SO→delivery→stock→invoice หากตัดสต็อกล้มเหลว ต้อง compensate (delivery กลับเป็น PENDING)

RULE · MONEY TYPE

จำนวนเงินใช้ decimal หรือ integer (สตางค์) ห้ามใช้ float

RULE · PRICING RULE ENGINE

ราคาพิเศษ/ส่วนลดเป็น engine แยก (pricing rule) ไม่ใช่ field คงที่ — match ตามเงื่อนไข (สินค้า/กลุ่ม/ลูกค้า/ช่วงวันที่/จำนวนขั้นต่ำ) แล้วเลือกกฎที่ priority สูงสุด · ราคาที่ได้ต้อง snapshot ลงเอกสาร

RULE · TAX PER LINE

คำนวณภาษี รายบรรทัด (per-line) ตาม item tax template แล้วค่อย aggregate หัวเอกสาร — รองรับหลายอัตรา/ยกเว้นภาษีในใบเดียว · เก็บ item-wise tax detail ไว้ตรวจสอบ

RULE · BUNDLE = PACKED ITEM

เมื่อขายสินค้า bundle ให้ explode เป็น packed_items เก็บ denormalized บนเอกสาร (SO/Delivery) เพื่อส่งให้ inventory ตัดสต็อกรายสินค้าย่อย และคง traceability
