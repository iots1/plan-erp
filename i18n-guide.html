มาตรฐานการออกแบบและการเขียนโค้ดเพื่อรองรับข้อมูล 2 ภาษา ครอบคลุมตั้งแต่ Database, Backend (NestJS) จนถึง Frontend โดยใช้หลักการ “จัดเก็บแบบแยกคอลัมน์ (Flat) แต่ส่งออกเป็นออบเจกต์ (Nested Object)” เพื่อให้ API ใช้งานง่ายและโค้ดสะอาดที่สุด

01 — Design Principle

หลักการออกแบบ

เราจะใช้หลักการ Suffix _th และ _en ในการแยกคอลัมน์เสมอ (ไม่ใช้ JSONB สำหรับฟิลด์ทั่วไป) เพื่อให้ได้ประสิทธิภาพสูงสุดในการทำ Index และ Full-Text Search — แต่เพื่อไม่ให้ Frontend ต้องยุ่งกับคอลัมน์คู่ขนาน Backend จะยุบทั้งสองคอลัมน์ให้เป็น Nested Object เดียวกันเสมอผ่าน Interceptor ก่อนส่งออก

DATABASE

Flat Columns

description_th, description_en — แยกคอลัมน์จริงในตาราง

BACKEND

Interceptor

รวบคู่คอลัมน์เป็น Object กลาง Response ทุก Endpoint โดยอัตโนมัติ

FRONTEND

Nested Object

patient.title?.th — อ่านค่าตรงจาก Object ไม่ต้องต่อ String Key

02 — Database & Entity (TypeORM)

การออกแบบ Database และ Entity

กรณี 1 — Master Data

Code + Text ที่แปลแล้ว

เช่น คำนำหน้า, สิทธิการรักษา — เก็บทั้ง title_code, title_th, title_en เพื่อลดการ Join ตอนอ่าน

กรณี 2 — Free Text

ผู้ใช้พิมพ์เอง

เช่น หมายเหตุ, รายละเอียด — เก็บแยกคอลัมน์ตรง ๆ description_th / description_en

ตัวอย่าง · Patient Entity
import { Entity, Column } from 'typeorm';

@Entity('patients')
export class Patient {
  // 1. กรณี Master Data: เก็บทั้ง Code และ Text ที่แปลแล้ว (เพื่อลดการ Join)
  @Column({ nullable: true })
  title_code: string;

  @Column({ nullable: true })
  title_th: string;

  @Column({ nullable: true })
  title_en: string;

  // 2. กรณี Free-text (ผู้ใช้พิมพ์เอง): เก็บแยกคอลัมน์
  @Column({ type: 'text', nullable: true })
  description_th: string;

  @Column({ type: 'text', nullable: true })
  description_en: string;
}
03 — Backend (NestJS) · Request Payload

ข้อมูลขาเข้า (Request)

การรับข้อมูลจาก Frontend (ผ่าน DTO) แบ่งเป็น 2 กรณีตามชนิดข้อมูล

Master Data

เช่น คำนำหน้า, สิทธิการรักษา — Frontend ส่งมาแค่ title_code หน้าที่หาคำแปล ไทย/อังกฤษเป็นของ Backend (ไปดึงจาก Master Data Microservice แล้วเซฟลง _th, _en)

Free Text

เช่น หมายเหตุ, รายละเอียด — Frontend ต้องส่งมาเป็น Flat Key ตาม Database คือ description_th และ description_en

04 — Backend (NestJS) · Response Payload

ข้อมูลขาออก (Response)

เพื่อไม่ให้ Frontend ต้องเขียนโค้ดที่รกด้วยการต่อ String (key + '_th') Backend จะทำการ ยุบคอลัมน์คู่ขนานให้กลายเป็น Object เสมอ ผ่าน Interceptor

ตัวอย่าง API Response ที่ Frontend จะได้รับ
{
  "id": "uuid-1234",
  "title_code": "01",
  "title": {
    "th": "นาย",
    "en": "Mr."
  },
  "description": {
    "th": "ปวดหัว",
    "en": null
  },
  "remark": {
    "th": null,
    "en": null
  }
}

สังเกตว่าจะมีการส่งค่า { th: null, en: null } ออกไปเสมอ แม้ข้อมูลใน DB จะเป็น Null เพื่อรักษา API Contract ให้คงที่

Interceptor · รวบฟิลด์ (Global หรือระดับ Controller)
import { Injectable, NestInterceptor, ExecutionContext, CallHandler } from '@nestjs/common';
import { Observable } from 'rxjs';
import { map } from 'rxjs/operators';

@Injectable()
export class LocalizationInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    return next.handle().pipe(
      map((data) => this.transformToLocaleObject(data))
    );
  }

  private transformToLocaleObject(data: any): any {
    if (data === null || typeof data !== 'object' || data instanceof Date) return data;
    if (Array.isArray(data)) return data.map(item => this.transformToLocaleObject(item));

    const transformed = {};
    const processedKeys = new Set<string>();

    for (const key in data) {
      if (processedKeys.has(key)) continue;

      // ตรวจจับคู่ _th และ _en
      if (key.endsWith('_th') || key.endsWith('_en')) {
        const baseKey = key.replace(/(_th|_en)$/, '');

        // รวบเป็น Object เดียวกัน และบังคับโครงสร้างแม้อีกภาษาจะไม่มีค่า
        transformed[baseKey] = {
          th: data[`${baseKey}_th`] ?? null,
          en: data[`${baseKey}_en`] ?? null
        };

        processedKeys.add(`${baseKey}_th`);
        processedKeys.add(`${baseKey}_en`);
      } else {
        transformed[key] = this.transformToLocaleObject(data[key]);
      }
    }
    return transformed;
  }
}
05 — Frontend (React / Angular / Vue) · การแสดงผล

View Mode

เนื่องจาก Backend ส่งข้อมูลมาเป็น Object เสมอ แต่ต้องระวังกรณีสุดวิสัยที่ Backend อาจส่งผิดพลาด ให้ Frontend ใช้ ?. (Optional Chaining) และ ?? (Nullish Coalescing) เสมอ

การเรียกใช้ที่ถูกต้อง
const currentLang = 'th'; // หรือรับจาก Context/Store

// จะคืนค่า '' แทนที่จะขึ้น undefined หรือทำให้แอปพัง
const displayTitle = patient.title?.[currentLang] ?? '';
const displayDesc = patient.description?.[currentLang] ?? '';
Helper Function กลาง · แนะนำให้ใช้ทั่วทั้งแอป
// utils/localization.util.ts
export type LocalizedString = { th: string | null; en: string | null };

export const getLocaleText = (
  localeObj: LocalizedString | null | undefined,
  lang: 'th' | 'en'
): string => {
  if (!localeObj) return '';
  // คืนค่าภาษาที่เลือก -> Fallback ไปอีกภาษา -> คืนค่าว่าง
  return localeObj[lang] ?? localeObj[lang === 'th' ? 'en' : 'th'] ?? '';
};

การใช้งานในหน้า UI:

<span>{{ getLocaleText(patient.title, currentLang) }}</span>
<p>{{ getLocaleText(patient.description, currentLang) }}</p>
