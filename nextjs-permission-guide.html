01 — ภาพรวม

สิ่งที่ต้องรู้ก่อนเริ่ม

ระบบสิทธิ์เป็น RBAC → Policy → Statement (ดูรายละเอียดเต็มที่ SRS Phase 1) — Next.js ไม่ต้อง implement ตรรกะ resolve สิทธิ์เอง แค่เรียก API ที่ถูกต้องแล้วเชื่อผลลัพธ์ที่ backend ส่งกลับมา

01

JWT ไม่มี permissions แล้ว

หลัง refactor เรื่อง session (§10 ใน SRS P1) permissions/roles ไม่ได้อยู่ใน JWT อีกต่อไป — ห้าม decode token ฝั่ง client เพื่อหาสิทธิ์ ต้องเรียก endpoint แทน

02

แหล่งสิทธิ์ที่ถูกต้อง

GET /auth/v1/auth/me คืน { roles, permissions, … } จาก session ฝั่ง server (Redis) — เป็นแหล่งความจริงเดียวที่ frontend ควรใช้

03

FE = UX เท่านั้น

ทุกอย่างที่ทำในหน้านี้ (ซ่อนปุ่ม/ซ่อน route) คือ UX — backend endpoint ทุกตัวยังต้องมี @RequirePermission() ตรวจซ้ำเสมอ 

sequenceDiagram
    participant NextJS as Next.js app
    participant AuthBC as auth-bc
    participant Redis as Redis · session:<jti>
    participant API as protected BC API (เช่น iam-bc)

    NextJS->>AuthBC: POST /auth/v1/auth/login { username, password }
    AuthBC->>Redis: เขียน session:<jti> (roles + permissions)

02 — Step 1

ประกาศ Permission ใหม่ผ่าน Manifest

ก่อนเขียนโค้ด Next.js สักบรรทัด ต้องประกาศ page:* / component:* ที่หน้าใหม่จะใช้เข้า catalog ก่อน — เพราะ JSX ไม่เหลือ literal string ให้ regex scan หลัง build (ต่างจากหน้า EJS ที่ scan data-permission="..." ได้ตรง ๆ)

ข้อจำกัด · ไฟล์ manifest ต้องอยู่ใน erp-api repo

script sync มองหาไฟล์ที่ apps/<service>/ui-permissions.manifest.json ภายใน erp-api repo (backend monorepo) เท่านั้น — ถ้า Next.js อยู่คนละ repo จริง ให้มีโฟลเดอร์บาง ๆ เช่น apps/frontend-web/ ใน erp-api repo ไว้ host แค่ไฟล์นี้ไฟล์เดียว (โค้ด Next.js จริงยังอยู่ repo เดิม)

03 — Step 2

ผูกสิทธิ์เข้า Policy → Role

ขั้นตอนนี้ ไม่ใช่งานของทีม Next.js — permission ที่เพิ่งเข้า catalog จาก Step 1 ยังไม่มีผลกับใครเลยจนกว่า Security Admin จะไปแนบมันเข้า policy แล้วเอา policy ไป attach เข้า role ผ่านหน้า Policy Generator (ดู SRS P1 §02 resolution chain / §09)

1 · User

มี role (เช่น ROLE_REPORT_VIEWER)

2 · Role → Policy

Security Admin แนบ policy ที่มี statement อนุญาต page:view_reports เข้า role นั้น

3 · Login ครั้งถัดไป

session ใหม่ (หรือ sync ทันทีถ้าใช้ §10) มี page:view_reports อยู่ใน permissions แล้ว

สรุปสั้น ๆ

"ประกาศ permission" (Step 1) ≠ "อนุญาตให้ใครใช้" (Step 2) — สองงานนี้แยกกันเสมอ และ Step 2 เป็นงานของ Security Admin ไม่ใช่ dev

04 — Step 3

Login แล้วดึงสิทธิ์จริงจาก /auth/me

Login คืนแค่ token — ไม่มี roles/permissions ติดมาด้วย (ย้ายไปอยู่ Redis session แล้ว) ต้องเรียก /auth/me อีกครั้งเพื่อได้ชุดสิทธิ์จริงมาเก็บไว้ใช้ทั้งแอป

RULE · เก็บ token ที่ฝั่ง client ให้เหมาะสม

endpoint ใช้ Bearer token ไม่ใช่ cookie — จะเก็บใน memory/sessionStorage ฝั่ง client component หรือทำ Route Handler เป็น BFF proxy ให้เก็บเป็น httpOnly cookie ก็ได้ ขึ้นกับ threat model ของทีม (ตัวอย่างในคู่มือนี้ใช้ sessionStorage เพื่อความง่ายในการอธิบาย logic การเช็คสิทธิ์เป็นหลัก)

อ้างอิง · มี implementation จริงของฝั่ง httpOnly cookie แล้ว

ถ้าทีมเลือกทาง BFF/cookie แทน sessionStorage — iam-view (admin UI ของ erp-api เอง) implement แบบนี้อยู่แล้วจริง พร้อม CSRF double-submit cookie protection (เพราะ cookie ถูกแนบอัตโนมัติ ต่างจาก Bearer header) ดูโค้ดอ้างอิงและ sequence diagram เต็มที่ SRS P1 §11 CSRF & Cookie Auth — access_token/csrf_token ที่ auth-bc ส่งกลับตอน login/refresh ใช้ได้กับทั้ง 2 flow พร้อมกันโดยไม่ต้องเปลี่ยน backend

05 — Step 4

Auth Context + usePermission Hook

ห่อทั้งแอปด้วย Context เดียวที่เก็บ session (รวม permissions) ไว้ในหน่วยความจำ แล้ว expose hasPermission() ให้ component อื่นเรียกใช้ — เขียนครั้งเดียว ใช้ได้ทั้งแอป

lib/auth/auth-context.tsx
'use client';

import { createContext, useContext, useEffect, useMemo, useState } from 'react';
import type { ReactNode } from 'react';

interface IUserSession {
  id: string | null;
  username: string | null;
  fullname: string | null;
  email: string | null;
  roles: string[];
  permissions: string[];
  jti: string | null;
}

interface IAuthContextValue {
  session: IUserSession | null;
  isLoading: boolean;
  hasPermission: (permission: string) => boolean;
  login: (username: string, password: string) => Promise<void>;
  logout: () => Promise<void>;
  // resync session (roles/permissions) จาก server — ดู §Step 4.5 เรื่อง stale cache
  refreshSession: () => Promise<IUserSession | null>;
}

const AuthContext = createContext<IAuthContextValue | null>(null);

// เช่น http://localhost:3001/auth/v1
const AUTH_BASE = process.env.NEXT_PUBLIC_AUTH_API_URL!;

async function fetchMe(accessToken: string): Promise<IUserSession> {
  const res = await fetch(`${AUTH_BASE}/auth/me`, {
    headers: { Authorization: `Bearer ${accessToken}` },
  });
  if (!res.ok) throw new Error('session expired');
  return res.json(); // raw JSON — /auth controller ไม่มี @ResourceType
}

export function AuthProvider({ children }: { children: ReactNode }) {
  const [session, setSession] = useState<IUserSession | null>(null);
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {
    const token = sessionStorage.getItem('access_token');
    if (!token) { setIsLoading(false); return; }
    fetchMe(token)
      .then(setSession)
      .catch(() => sessionStorage.removeItem('access_token'))
      .finally(() => setIsLoading(false));
  }, []);

  const login = async (username: string, password: string) => {
    const res = await fetch(`${AUTH_BASE}/auth/login`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ username, password }),
    });
    if (!res.ok) throw new Error('invalid credentials');
    const { access_token, refresh_token } = await res.json();
    sessionStorage.setItem('access_token', access_token);
    sessionStorage.setItem('refresh_token', refresh_token);
    setSession(await fetchMe(access_token));
  };

  const logout = async () => {
    const token = sessionStorage.getItem('access_token');
    if (token) {
      await fetch(`${AUTH_BASE}/auth/logout`, {
        method: 'POST',
        headers: { Authorization: `Bearer ${token}`, 'Content-Type': 'application/json' },
        body: JSON.stringify({ refresh_token: sessionStorage.getItem('refresh_token') }),
      });
    }
    sessionStorage.clear();
    setSession(null);
  };

  // permissions มาจาก /auth/me เท่านั้น — ไม่ decode JWT เอง
  const hasPermission = (permission: string) =>
    session?.permissions.includes(permission) ?? false;

  // เรียกซ้ำเมื่อสงสัยว่าสิทธิ์เปลี่ยนกลางคัน (§Step 4.5) — ไม่ทำอะไรถ้ายังไม่ login
  const refreshSession = async (): Promise<IUserSession | null> => {
    const token = sessionStorage.getItem('access_token');
    if (!token) { setSession(null); return null; }
    try {
      const fresh = await fetchMe(token);
      setSession(fresh);
      return fresh;
    } catch {
      sessionStorage.removeItem('access_token');
      setSession(null);
      return null;
    }
  };

  const value = useMemo(
    () => ({ session, isLoading, hasPermission, login, logout, refreshSession }),
    [session, isLoading],
  );

  return <AuthContext.Provider value={value}>{children}</AuthContext.Provider>;
}

export function useAuth(): IAuthContextValue {
  const ctx = useContext(AuthContext);
  if (!ctx) throw new Error('useAuth must be used inside <AuthProvider>');
  return ctx;
}
RULE · Cache สิทธิ์ฝั่ง client เป็นสถานะ ณ ตอน login เท่านั้น

session ใน Context นี้เป็นแค่ snapshot ตอน login/refresh — ถ้า backend แก้ policy/role/permission กลางคันจะเกิดช่วงที่ client cache ไม่ตรงกับ server ดูวิธีจัดการเต็ม ๆ ที่ Step 4.5 ด้านล่าง

06 — Step 4.5

เมื่อ Backend แก้ Policy/Role/Permission กลางคัน

คำถามที่พบบ่อย: "ฝั่ง Next.js เก็บ permissions ไว้ตอน login ครั้งแรก ถ้า admin ไปแก้ policy/role/permission ทีหลัง ฝั่ง frontend จะเป็นยังไง" — คำตอบสั้น ๆ คือ server รู้ทันที (§10 Session Sync ของ SRS P1) แต่ client ที่ cache session ไว้ใน memory จะไม่รู้เอง จนกว่าจะ refetch — ส่วนนี้อธิบายช่วงที่ไม่ตรงกันนี้และวิธีจัดการ

สิทธิ์ถูกเพิ่มใหม่ปุ่ม/เมนู/หน้าที่เพิ่งได้สิทธิ์ยังไม่โผล่ไม่มี error — แค่ยังไม่เห็นของใหม่จนกว่าจะ refetch
logout/revoke session ทั้งหมด (admin force logout)UI ยังค้างเหมือน login อยู่เรียก API ใดก็ตามจะโดน 401 (session หายจาก Redis) ไม่ใช่ 403

3 ชั้นการจัดการ — เรียงจาก "ต้องมี" ไป "เสริมให้ UX ดีขึ้น"

1 · Reactive (ต้องมี)
Global 403 handler

ทุก API call ผ่าน wrapper เดียว — เจอ 403 ปุ๊บ เรียก refreshSession() ทันที รับประกันว่า UI จะตรงกับ server เสมอภายใน 1 request หลังจากพลาดครั้งแรก

2 · Periodic (แนะนำ)
Refetch on focus/interval

react-query/SWR ตั้ง refetchOnWindowFocus + refetchInterval — ลดโอกาสที่ user จะเจอ 403 ตั้งแต่แรกโดยไม่ต้องรอ error

3 · On-navigation (ทางเลือก)
Refetch เมื่อเปลี่ยนหน้า

เรียก refreshSession() ทุกครั้งก่อน <PermissionGate> ตรวจ requires ของหน้าใหม่ — จำกัดความ stale ไว้แค่ "ระหว่างอยู่หน้าเดิม" ไม่ลามข้ามหน้า

lib/api-client.ts · global fetch wrapper (ชั้นที่ 1 — ต้องมี)
'use client';

import { useCallback } from 'react';
import { useRouter } from 'next/navigation';
import { useAuth } from './auth/auth-context';

interface IApiErrorBody {
  status: { code: number; message: string };
}

export function useApiClient() {
  const { refreshSession, logout } = useAuth();
  const router = useRouter();

  const request = useCallback(async <T>(url: string, init: RequestInit = {}): Promise<T> => {
    const token = sessionStorage.getItem('access_token');
    const res = await fetch(url, {
      ...init,
      headers: {
        'Content-Type': 'application/json',
        ...(token ? { Authorization: `Bearer ${token}` } : {}),
        ...init.headers,
      },
    });

    // session หายไปเลย (401) เช่น admin force-logout — ต้อง login ใหม่ ไม่ใช่แค่ refetch
    if (res.status === 401) {
      await logout();
      router.replace('/login');
      throw new Error('unauthenticated');
    }

    // 403 = permission ถูกปฏิเสธ ณ ตอนนี้จริง (server เพิ่ง sync ล่าสุดแล้ว) —
    // resync ให้ Context ตรงกับ server ทันที ก่อนพาไปหน้า 403
    if (res.status === 403) {
      await refreshSession();
      const body = (await res.clone().json().catch(() => null)) as IApiErrorBody | null;
      router.push(`/403?reason=${encodeURIComponent(body?.status.message ?? 'forbidden')}`);
      throw new Error(body?.status.message ?? 'forbidden');
    }

    if (!res.ok) throw new Error(`request failed: ${res.status}`);
    return res.json();
  }, [refreshSession, logout, router]);

  return { request };
}

รูปแบบ error body ด้านบน (status.code) มาจาก AllExceptionsFilter จริง — ForbiddenException ทั่วไปที่ PermissionGuard โยนจะได้ status.code = 403 (เลขธรรมดา ไม่ใช่ 6 หลักแบบ 400001/400002 ที่สงวนไว้เฉพาะ ValidationException/InvalidParameterException)

ชั้นที่ 2 (เสริม) · refetch on focus/interval ด้วย react-query
import { useQuery } from '@tanstack/react-query';

function useSessionQuery(accessToken: string | null) {
  return useQuery({
    queryKey: ['session', accessToken],
    queryFn: () => fetchMe(accessToken!),
    enabled: !!accessToken,
    refetchOnWindowFocus: true,  // สลับกลับมาที่ tab ก็ resync
    refetchInterval: 60_000,      // สำรอง เผื่อไม่เคยสลับ tab เลย
  });
}
RULE · ไม่ต้องเชื่อ client cache เพื่อความปลอดภัย เชื่อแค่เพื่อ UX

ไม่ว่าฝั่ง Next.js จะ refetch บ่อยแค่ไหน PermissionGuard ฝั่ง backend อ่านจาก Redis สดใหม่ทุก request อยู่แล้ว (§10 Session Sync) — 3 ชั้นด้านบนมีไว้แก้ปัญหา UX ที่ปุ่ม/หน้าโชว์ผิด ไม่ใช่ช่องโหว่ความปลอดภัยที่ต้องอุด

07 — Step 5

สร้าง Route ใหม่ + Page Guard

Component กลาง <PermissionGate> เช็ค 1 permission ระดับหน้า — ไม่มีสิทธิ์ → เด้งไปหน้า 403 (หรือ login ถ้ายังไม่ได้ authenticate)

components/permission-gate.tsx
'use client';

import { useEffect } from 'react';
import { useRouter } from 'next/navigation';
import { useAuth } from '@/lib/auth/auth-context';
import type { ReactNode } from 'react';

export function PermissionGate({
  requires,
  children,
}: {
  requires: string;
  children: ReactNode;
}) {
  const { session, isLoading, hasPermission } = useAuth();
  const router = useRouter();

  useEffect(() => {
    if (isLoading) return;
    if (!session) { router.replace('/login'); return; }
    if (!hasPermission(requires)) { router.replace('/403'); }
  }, [isLoading, session, requires]);

  // กัน flash ของเนื้อหาก่อน redirect เสร็จ
  if (isLoading || !session || !hasPermission(requires)) return null;
  return <>{children}</>;
}

  );
}
RULE · middleware.ts เช็คได้แค่ "login แล้วหรือยัง"

Edge middleware รันก่อนทุก request และไม่ควรยิง network call ไปเช็ค permission ทุกครั้ง (ช้า + เพิ่มโหลด auth-bc) — ให้ middleware เช็คแค่ "มี token ไหม" (redirect ไป /login ถ้าไม่มี) ส่วนการเช็ค permission รายหน้า ให้เป็นหน้าที่ของ <PermissionGate> ที่ทำงานหลัง Context โหลด session (จาก cache ใน memory ไม่ใช่ network call ใหม่ทุกครั้ง)

08 — Step 6

ซ่อน/แสดง Component ย่อยด้วย <Can>

ปุ่ม/วิดเจ็ตที่ต้องซ่อนเป็นรายชิ้น (ไม่ใช่ทั้งหน้า) ใช้ component เล็ก ๆ ตัวเดียวห่อไว้ — เทียบเท่ากับ data-permission ฝั่ง EJS

components/can.tsx
'use client';

import { useAuth } from '@/lib/auth/auth-context';
import type { ReactNode } from 'react';

export function Can({
  permission,
  children,
  fallback = null,
}: {
  permission: string;
  children: ReactNode;
  fallback?: ReactNode;
}) {
  const { hasPermission } = useAuth();
  return hasPermission(permission) ? <>{children}</> : <>{fallback}</>;
}

09 — Step 7

Backend Guard ยังต้องมีเสมอ

ขั้นตอนนี้ไม่ใช่งาน Next.js แต่เป็นสิ่งที่ทีม backend (BC ที่เป็นเจ้าของ endpoint จริง เช่น reports-bc) ต้องทำคู่กันเสมอ — ไม่งั้น <PermissionGate>/<Can> ที่เพิ่งเขียนจะเป็นแค่ล็อกกระดาษ

Next.js (UX เท่านั้น)

<PermissionGate> / <Can> ซ่อน route/ปุ่มจากผู้ใช้ที่ไม่ควรเห็น — แก้ query string/inspect element ผ่านได้ ไม่ใช่มาตรการความปลอดภัย

Backend Guard (Security จริง)

@RequirePermission('report:view') ที่ controller ของ endpoint จริง — AuthGuard/PermissionGuard ตรวจทุก request ไม่สนว่าเรียกจาก Next.js, EJS, curl หรือ Postman

RULE · FRONTEND ≠ SECURITY (ครอบคลุมทุก frontend)

กฎนี้ (SRS P1 §05) ใช้กับทุก frontend เหมือนกันหมด ไม่ว่าจะเป็น EJS วันนี้หรือ Next.js — ทุก request ต้องถูกตรวจซ้ำที่ backend Guard เสมอ
