Identity & Access — Login · RBAC · Policy-Based Access (PBAC/ABAC)

รากฐานความปลอดภัย — พิสูจน์ตัวตน (JWT + refresh + Redis) และควบคุมสิทธิ์แบบ Policy (Effect · Resource · Action · Condition) ครอบทั้งฝั่ง UI และ API · อ้างอิงจาก prototype user-management.html

01

Business Overview & Roles · ใครทำอะไร

SECURITY ADMIN
ผู้ดูแลความปลอดภัย

สร้าง Role / Policy (Policy Generator), แนบ policy เข้า role, กำหนดนโยบายรหัสผ่าน/บล็อกผู้ใช้

HR / USER ADMIN
ฝ่ายบุคคล

สร้างบัญชีผู้ใช้ (พนักงาน), กำหนดแผนก และ assign role ให้ผู้ใช้

END USER
ผู้ใช้งานระบบ

login เข้าระบบ 2 ภาษา · เห็นเมนู/ปุ่ม/หน้าจอเฉพาะที่มีสิทธิ์ (สิทธิ์มาจาก JWT)

auth-bc · db: auth

พิสูจน์ตัวตน — credentials, ออก/ตรวจ JWT + refresh, session ใน Redis, ประวัติ login, บล็อกผู้ใช้, security log

iam-bc · db: iam

แหล่งความจริงของ Users · Roles · Policies · Permissions — เป็นเจ้าของสิทธิ์ทั้งหมด (RBAC + PBAC)

วัตถุประสงค์ของเฟส: ทุกบริการหลัง gateway ต้อง ตรวจสิทธิ์จาก iam ก่อนทำงาน · ชื่อ role/permission ที่แสดงผลเก็บ 2 ภาษาแบบ flat column (name_th/name_en) และทุกตารางสืบทอด BaseEntity ตาม Backend Conventions
02

Access Control Model · RBAC → Policy → Statement

ระบบใช้ RBAC (สิทธิ์ผ่านบทบาท) ต่อยอดด้วย Policy-Based (สิทธิ์เป็นเอกสาร statement แบบ AWS IAM) และรองรับ ABAC (เงื่อนไขตาม attribute) — Deny ชนะ Allow เสมอ

การไล่สิทธิ์ (Resolution chain)
User ─ user_roles → Role ─ role_policies → Policy → Statement[] → Permission (resource:action)

ตอน login ระบบ รวม (flatten) ทุก statement ของทุก policy ในทุก role → ได้ชุด permission สุทธิ ใส่ลงใน JWT · ฝั่ง UI และ API ใช้ชุดนี้ตรวจสิทธิ์

กายวิภาคของ Statement · 1 กฎประกอบด้วย

ส่วนตัวอย่างค่าคำอธิบาย
effectallow | denyอนุญาต/ปฏิเสธ — deny ชนะเสมอ (deny-override)
servicefrontend-ui · inventory-bc · system-adminระบบ/บริการเป้าหมาย (UI ใช้ implicit frontend-ui)
resourcegoods_receipt · user_account · page_dashboardทรัพยากรที่ถูกเข้าถึง · * = ทั้งหมด
actionsgoods_receipt:submit · page:view_usersการกระทำ (permission รูป resource:action)
conditions{ StringEquals: { … } }เงื่อนไข ABAC (optional) — จริงถึงจะมีผล
UI plane

คุมสิ่งที่ผู้ใช้ เห็น/กดได้ — page:* (เข้าหน้า) และ component:* (ปุ่ม/กล่อง/วิดเจ็ต)

API plane

คุมสิ่งที่ผู้ใช้ เรียก backend ได้ — <resource>:<action> เช่น invoice:approve ตรวจที่ NestJS Guard

04

Use Cases & Flows · กระบวนการหลัก

UC-1.1 · LOGIN (JWT + Refresh + Redis)
1

ส่ง username + password → auth ตรวจกับ credentials (bcrypt/argon2)

2

iam รวม permission สุทธิของผู้ใช้ (ทุก role → policy → statement)

3

ออก access token (อายุสั้น, มี permissions) + refresh token · เก็บ session ใน Redis

4

บันทึก login_histories · ผิดเกินกำหนด → blocked_users

UC-1.2 · AUTHORIZATION (ทุก request)
1

Gateway/Guard ตรวจ access token (ลายเซ็น + หมดอายุ + session ใน Redis)

2

Guard ต้องการ resource:action ของ endpoint → เช็คกับ permissions ใน JWT

3

ถ้ามี condition → ประเมินกับ context (เจ้าของข้อมูล/สาขา/เวลา/IP)

4

deny ชนะ allow · ไม่ผ่าน → 403 Forbidden

UC-1.3 · MANAGEMENT SCREENS (จาก prototype)

Users — สร้าง/แก้บัญชี, กำหนดแผนก, assign role (user_account:assign_role)

Roles — สร้าง role, แนบ policies (attach)

Policies — รายการนโยบายทั้งหมด

Policy Generator — สร้าง statement: เลือก UI/API → Service → Resource → Actions → Conditions

SUBMITTABLE-LIKE · REVOCATION

การเปลี่ยน role/policy ต้อง ทำให้ session เดิม refresh สิทธิ์ใหม่ (เพิกถอน refresh token หรือ re-issue) เพื่อไม่ให้สิทธิ์เก่าค้างใน JWT · ทุกการแก้สิทธิ์บันทึก security_logs

Policy Examples & Enforcement · ตัวอย่าง + การบังคับใช้

รูปแบบ JSON ตรงกับผลลัพธ์ของ Policy Generator ใน prototype (compile condition เป็น { operator: { key: value } })

UI A · ผู้จัดการเข้าหน้าแดชบอร์ด (เห็นบางกล่อง) 
{
  "code": "POL_MANAGER_DASHBOARD", "name_th": "ผู้จัดการ · แดชบอร์ด",
  "statements": [{
    "effect": "allow", "plane": "ui", "service": ["frontend-ui"],
    "resource": ["page_dashboard"],
    "actions": ["page:view_dashboard", "component:widget_total_users", "component:widget_total_roles"]
  }]
}

→ เห็นหน้า Dashboard + 2 กล่องสถิติ · ไม่เห็นกราฟ (ไม่มี component:widget_login_graph)

API B · แอดมินจัดการผู้ใช้ (assign role) 
{
  "code": "POL_ADMIN_USER_MGMT",
  "statements": [{
    "effect": "allow", "plane": "api", "service": ["system-admin"],
    "resource": ["user_account"],
    "actions": ["user_account:read", "user_account:create", "user_account:assign_role"]
  }]
}
// เลือก "ทั้งหมด" ในหน้า Generator → resource: ["*"] (wildcard)
ABAC + DENY C · พนักงานขายเห็นเฉพาะลูกค้าของตน · ห้ามลบนอกเวลา 
{
  "code": "POL_SALES_OWN_CUSTOMER",
  "statements": [
    { "effect": "allow", "plane": "api", "service": ["sales-bc"],
      "resource": ["customer"], "actions": ["customer:read", "customer:edit"],
      "conditions": { "StringEquals": { "customer.owner_id": "${user.id}" } } },   // เฉพาะลูกค้าของตัวเอง
    { "effect": "deny", "plane": "api", "service": ["sales-bc"],
      "resource": ["customer"], "actions": ["customer:delete"],
      "conditions": { "DateGreaterThan": { "context.current_time": "18:00" } } }        // ห้ามลบหลัง 18:00 (deny ชนะ)
  ]
}

การบังคับใช้ 2 ฝั่ง

FRONTEND · data-permission

ทุกปุ่ม/เมนู/วิดเจ็ตแนบ data-permission — ถ้า JWT ไม่มีสิทธิ์ → ซ่อน (no-permission) · หมวดเมนูที่ไม่มีลูกที่มองเห็นจะถูกซ่อน · route guard กันเข้าหน้า

<button data-permission="component:btn_add_user">
  เพิ่มบุคลากร
</button>
BACKEND · NestJS Guard

endpoint ประกาศ permission ที่ต้องมี · Guard เช็คกับ JWT + ประเมิน condition · deny-override

@RequirePermission('goods_receipt:submit')
@Post(':id/submit')
submit(@Param('id') id: string) { … }
FRONTEND ≠ SECURITY

การซ่อนปุ่มฝั่ง UI เป็นเรื่อง UX เท่านั้น — ทุก request ต้องถูกตรวจซ้ำที่ backend Guard เสมอ (อย่าเชื่อ client)

07

Developer Implementation Guide · คำสั่งจาก SA

RULE · TOKEN + SESSION

access token อายุสั้น (เช่น 15 นาที) แนบทุก request · refresh token อายุยาวเก็บ hash ใน DB · session/blacklist ใน Redis · logout/เปลี่ยนสิทธิ์ = เพิกถอน refresh token

RULE · DENY-OVERRIDE

รวมสิทธิ์แบบ AWS IAM — ถ้ามี deny ตรงกับ resource:action ใด ๆ → ปฏิเสธทันที แม้จะมี allow · ไม่มี allow = ปฏิเสธโดยปริยาย (default deny)

RULE · CONDITION EVAL

ประเมิน condition ที่ backend กับ context จริง (${user.id}, ${user.department}, IP, เวลา) · operator รองรับ StringEquals/StringLike/Numeric*/Date*/IpAddress

RULE · GUARD ทุก endpoint

ทุก endpoint ที่ไม่ public ต้องมี @RequirePermission('resource:action') · frontend data-permission เป็น UX เท่านั้น ห้ามใช้แทนการตรวจ backend

RULE · PASSWORD

เก็บเฉพาะ hash (bcrypt/argon2) ไม่เก็บ plaintext · ใส่ผิดเกินกำหนด → เพิ่ม blocked_users · ทุกเหตุการณ์ลง security_logs

RULE · i18n + BASE ENTITY

ชื่อ role/policy/permission ที่แสดงผลเป็น flat column *_th / *_en ยุบเป็น object ตอน response · ทุกตารางสืบทอด BaseEntity (audit + soft-delete)

RULE · UUID + SOURCE OF TRUTH

iam เป็นเจ้าของ users · auth และบริการอื่นอ้าง user_id เป็น UUID · ห้าม FK ข้าม DB · เปลี่ยนสิทธิ์ยิง event ให้บริการ cache สิทธิ์ล้าง/รีเฟรช

08

Glossary · อภิธานศัพท์ IAM / Security

ศัพท์ความหมาย
Authentication (AuthN)การพิสูจน์ตัวตน — "คุณคือใคร" (login ด้วยรหัสผ่าน)
Authorization (AuthZ)การตรวจสิทธิ์ — "คุณทำอะไรได้บ้าง"
RBACRole-Based Access Control — ให้สิทธิ์ผ่านบทบาท (role)
PBACPolicy-Based — สิทธิ์เป็นเอกสาร policy (หลาย statement) แบบ AWS IAM
ABACAttribute-Based — สิทธิ์ขึ้นกับ attribute/เงื่อนไข (เช่น เจ้าของข้อมูล · สาขา · เวลา)
Policyชุดกฎสิทธิ์ที่แนบเข้า role — ประกอบด้วยหลาย statement
Statementกฎ 1 ข้อ = Effect + Service + Resource + Action + Condition
Effectผลของกฎ — allow หรือ deny
Permissionสิทธิ์ย่อยรูป resource:action (เช่น invoice:approve)
Resourceทรัพยากรที่ถูกเข้าถึง (goods_receipt · customer · page_dashboard)
Actionการกระทำต่อ resource (read · create · submit · delete)
Conditionเงื่อนไข ABAC = operator + key + value
Deny-overrideถ้ามี deny ตรงกัน → ปฏิเสธเสมอ แม้มี allow
Least Privilegeให้สิทธิ์เท่าที่จำเป็น (default deny)
UI plane / API planeสิทธิ์ฝั่งหน้าจอ (page/component) vs ฝั่ง API (resource/action)
JWTJSON Web Token — token ที่มี claims (sub · roles · permissions · exp)
Claimข้อมูลภายใน JWT
Access / Refresh Tokentoken อายุสั้นแนบทุก request / token อายุยาวใช้ขอ access token ใหม่
Guardตัวดักตรวจสิทธิ์ฝั่ง backend (NestJS) ก่อนเข้า handler
Route Guardตัวดักฝั่ง frontend กันเข้าหน้าที่ไม่มีสิทธิ์
Sessionสถานะการ login (เก็บใน Redis) — เพิกถอนได้
Wildcard (*)ครอบทุก resource/action ในกลุ่มนั้น
