02

Access Control Model · RBAC → Policy → Statement

ระบบใช้ RBAC (สิทธิ์ผ่านบทบาท) ต่อยอดด้วย Policy-Based (สิทธิ์เป็นเอกสาร statement แบบ AWS IAM) และรองรับ ABAC (เงื่อนไขตาม attribute) — Deny ชนะ Allow เสมอ

การไล่สิทธิ์ (Resolution chain)
User ─ user_roles → Role ─ role_policies → Policy → Statement[] → Permission (resource:action)

ตอน login ระบบ รวม (flatten) ทุก statement ของทุก policy ในทุก role → ได้ชุด permission สุทธิ ใส่ลงใน JWT · ฝั่ง UI และ API ใช้ชุดนี้ตรวจสิทธิ์

กายวิภาคของ Statement · 1 กฎประกอบด้วย

ส่วนตัวอย่างค่าคำอธิบาย
effectallow | denyอนุญาต/ปฏิเสธ — deny ชนะเสมอ (deny-override)
servicefrontend-ui · inventory-bc · system-adminระบบ/บริการเป้าหมาย (UI ใช้ implicit frontend-ui)
resourcegoods_receipt · user_account · page_dashboardทรัพยากรที่ถูกเข้าถึง · * = ทั้งหมด
actionsgoods_receipt:submit · page:view_usersการกระทำ (permission รูป resource:action)
conditions{ StringEquals: { … } }เงื่อนไข ABAC (optional) — จริงถึงจะมีผล
UI plane

คุมสิ่งที่ผู้ใช้ เห็น/กดได้ — page:* (เข้าหน้า) และ component:* (ปุ่ม/กล่อง/วิดเจ็ต)

API plane

คุมสิ่งที่ผู้ใช้ เรียก backend ได้ — <resource>:<action> เช่น invoice:approve ตรวจที่ NestJS Guard

Use Cases & Flows · กระบวนการหลัก

UC-1.1 · LOGIN (JWT + Refresh + Redis)
1

ส่ง username + password → auth ตรวจกับ credentials (bcrypt/argon2)

2

iam รวม permission สุทธิของผู้ใช้ (ทุก role → policy → statement)

3

ออก access token (อายุสั้น, มี permissions) + refresh token · เก็บ session ใน Redis

4

บันทึก login_histories · ผิดเกินกำหนด → blocked_users

UC-1.2 · AUTHORIZATION (ทุก request)
1

Gateway/Guard ตรวจ access token (ลายเซ็น + หมดอายุ + session ใน Redis)

2

Guard ต้องการ resource:action ของ endpoint → เช็คกับ permissions ใน JWT

3

ถ้ามี condition → ประเมินกับ context (เจ้าของข้อมูล/สาขา/เวลา/IP)

4

deny ชนะ allow · ไม่ผ่าน → 403 Forbidden

UC-1.3 · MANAGEMENT SCREENS (จาก prototype)

Users — สร้าง/แก้บัญชี, กำหนดแผนก, assign role (user_account:assign_role)

Roles — สร้าง role, แนบ policies (attach)

Policies — รายการนโยบายทั้งหมด

Policy Generator — สร้าง statement: เลือก UI/API → Service → Resource → Actions → Conditions

ตัวอย่างที่ใช้งานได้จริง (สำหรับ dev)

ทั้ง 4 หน้าจอนี้มี prototype คลิกได้จริงที่ prototype/user-management.html — รวม mock JWT simulator (สลับ role ดูว่าเมนู/ปุ่มไหนหาย), ตาราง Users/Roles/Policies และ Policy Generator ที่ compile statement ออกมาเป็น JSON จริงตามรูปแบบใน §05

SUBMITTABLE-LIKE · REVOCATION

การเปลี่ยน role/policy ต้อง ทำให้ session เดิม refresh สิทธิ์ใหม่ (เพิกถอน refresh token หรือ re-issue) เพื่อไม่ให้สิทธิ์เก่าค้างใน JWT · ทุกการแก้สิทธิ์บันทึก security_logs

การบังคับใช้ 2 ฝั่ง

FRONTEND · data-permission

ทุกปุ่ม/เมนู/วิดเจ็ตแนบ data-permission — ถ้า JWT ไม่มีสิทธิ์ → ซ่อน (no-permission) · หมวดเมนูที่ไม่มีลูกที่มองเห็นจะถูกซ่อน · route guard กันเข้าหน้า

<button data-permission="component:btn_add_user">
  เพิ่มบุคลากร
</button>
BACKEND · NestJS Guard

endpoint ประกาศ permission ที่ต้องมี · Guard เช็คกับ JWT + ประเมิน condition · deny-override

@RequirePermission('goods_receipt:submit')
@Post(':id/submit')
submit(@Param('id') id: string) { … }
FRONTEND ≠ SECURITY

การซ่อนปุ่มฝั่ง UI เป็นเรื่อง UX เท่านั้น — ทุก request ต้องถูกตรวจซ้ำที่ backend Guard เสมอ (อย่าเชื่อ client)

Developer Implementation Guide · คำสั่งจาก SA

RULE · TOKEN + SESSION

access token อายุสั้น (เช่น 15 นาที) แนบทุก request · refresh token อายุยาวเก็บ hash ใน DB · session/blacklist ใน Redis · logout/เปลี่ยนสิทธิ์ = เพิกถอน refresh token · แนบ token ได้ 2 วิธี — Authorization: Bearer (API/Next.js client) หรือ httpOnly cookie + CSRF double-submit (browser admin UI) — รายละเอียดเต็มที่ §11 CSRF & Cookie Auth

RULE · DENY-OVERRIDE

รวมสิทธิ์แบบ AWS IAM — ถ้ามี deny ตรงกับ resource:action ใด ๆ → ปฏิเสธทันที แม้จะมี allow · ไม่มี allow = ปฏิเสธโดยปริยาย (default deny)

RULE · CONDITION EVAL

ประเมิน condition ที่ backend กับ context จริง (${user.id}, ${user.department}, IP, เวลา) · operator รองรับ StringEquals/StringLike/Numeric*/Date*/IpAddress

RULE · GUARD ทุก endpoint

ทุก endpoint ที่ไม่ public ต้องมี @RequirePermission('resource:action') · frontend data-permission เป็น UX เท่านั้น ห้ามใช้แทนการตรวจ backend

RULE · PASSWORD

เก็บเฉพาะ hash (bcrypt/argon2) ไม่เก็บ plaintext · ใส่ผิดเกินกำหนด → เพิ่ม blocked_users · ทุกเหตุการณ์ลง security_logs

RULE · i18n + BASE ENTITY

ชื่อ role/policy/permission ที่แสดงผลเป็น flat column *_th / *_en ยุบเป็น object ตอน response · ทุกตารางสืบทอด BaseEntity (audit + soft-delete)

RULE · UUID + SOURCE OF TRUTH

iam เป็นเจ้าของ users · auth และบริการอื่นอ้าง user_id เป็น UUID · ห้าม FK ข้าม DB · เปลี่ยนสิทธิ์ยิง event ให้บริการ cache สิทธิ์ล้าง/รีเฟรช

Session Sync & Revocation · JWT ↔ Redis Session (Implemented)

ส่วนนี้บันทึกการ implement จริงที่แก้ปัญหา 2 ข้อจาก §07 RULE · UUID + SOURCE OF TRUTH ("เปลี่ยนสิทธิ์ยิง event ให้บริการ cache สิทธิ์ล้าง/รีเฟรช") — (1) JWT payload บวมขึ้นเรื่อย ๆ ตามจำนวน permission ที่ผู้ใช้มี และ (2) การถอนสิทธิ์ไม่มีผลจนกว่า access token จะหมดอายุ

ปัญหาเดิม

permissions/roles/conditional_permissions ฝังอยู่ใน JWT claims ตั้งแต่ตอน login — token โตขึ้นเรื่อย ๆ ตามจำนวนสิทธิ์ (เสี่ยงชน header size limit ของ proxy บางตัว) และสิทธิ์ที่ฝังไปแล้วแก้ไม่ได้จนกว่า token หมดอายุ (ปกติ 15 นาที)

แนวทางแก้

ย้าย permissions/roles ออกจาก JWT ไปเก็บใน Redis session blob (session:<jti>) ที่ auth-bc เขียนไว้ตั้งแต่ login — JWT เหลือแค่ claims ระบุตัวตน (คงที่ไม่ว่าจะมีกี่สิทธิ์) ส่วน iam-bc เขียนสิทธิ์ใหม่ทับ session เดิมได้ทันทีที่มีการเปลี่ยน role/policy โดยไม่ต้องบังคับ logout

CSRF & Cookie-Based Browser Auth · Double-Submit Cookie (Implemented)

ส่วนนี้บันทึกการ implement จริงของ iam-view (admin UI แบบ EJS) ที่ย้ายจากเก็บ access_token/refresh_token ใน localStorage ไปเป็น httpOnly cookie — client อื่น (Next.js, Postman, service-to-service) ยังใช้ Authorization: Bearer แบบเดิมได้ทุกอย่าง ไม่กระทบ (§09 AUTH + CORS)

ปัญหาเดิม

localStorage เข้าถึงได้จาก JavaScript ทุกตัวที่รันบนหน้านั้น — ถ้ามีช่องโหว่ XSS แม้แค่จุดเดียว ก็ขโมย access_token/refresh_token ออกไปได้ทันที (token theft)

แนวทางแก้

ย้าย token ไปเป็น httpOnly cookie (JS อ่านค่าไม่ได้เลย แม้จะมี XSS) — แต่ cookie ถูก browser แนบให้อัตโนมัติทุก request (ambient credential) จึงเปิดช่องโหว่ CSRF ใหม่แทน ต้องปิดคู่กันด้วย double-submit cookie

RULE · DOUBLE-SUBMIT ไม่ใช่ SERVER-SIDE LOOKUP

CsrfGuard (libs/common/src/guards/csrf.guard.ts) แค่เทียบ header กับ cookie ตรงกันไหม (timing-safe compare) — ไม่ query Redis/DB เพิ่ม เพราะความปลอดภัยของ pattern นี้อยู่ที่ attacker เว็บอื่นอ่าน/ตั้ง cookie ของ origin เราไม่ได้เลย (same-origin policy) ไม่ได้อยู่ที่การ verify ค่าย้อนกลับกับ server

RULE · เช็คจาก "มี cookie csrf_token ไหม" ไม่ใช่ "@Public() หรือเปล่า"

POST /auth/refresh เป็น @Public() (ไม่ต้องมี JWT ก็เรียกได้) แต่พอ refresh_token กลายเป็น cookie แล้ว มันก็ ambient เหมือนกัน — CsrfGuard จึงเช็คจากการมีอยู่ของ cookie csrf_token โดยตรง ไม่ได้อิงกับ @Public()/มี JWT หรือไม่ ยกเว้นแค่ POST /auth/login ที่มี @SkipCsrfCheck() ชัดเจน (ยังไม่มี cookie ให้เช็คตอนนั้น — ป้องกันด้วย credential ในตัว body แทน)

RULE · CORS_ORIGIN ห้ามเป็น "*" เมื่อใช้ cookie

Browser reject การรวมกันของ Access-Control-Allow-Origin: * กับ request ที่ส่ง credentials: 'include' เสมอ — ทุก BC ตั้ง credentials: true ไว้เป็นค่าเริ่มต้นใน bootstrap.util.ts แล้ว ดังนั้น deployment ไหนก็ตามที่มี browser client ต้องตั้ง env CORS_ORIGIN เป็น origin จริง (คั่นด้วย comma ได้ถ้ามีหลาย origin) ไม่ใช่ wildcard

RULE · ไม่มี DB/Redis schema เปลี่ยน

csrf_token เป็นค่าสุ่ม 256-bit ที่ generate ใหม่ทุก login/refresh (AuthService.issueTokens()) — ไม่ได้เก็บลง Postgres หรือ Redis session เพิ่ม เพราะ pattern นี้ verify แค่ header ↔ cookie เท่านั้น (ดู RULE ด้านบน) จึง deploy ได้โดยไม่ต้อง migration
