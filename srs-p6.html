Business Overview & Roles · ใครทำอะไร

EXECUTIVE
ผู้บริหาร

ดู Dashboard ยอดขาย, กำไรขั้นต้น, สินค้าขายดี, สต็อกใกล้หมด/หมดอายุ ในภาพรวม

DEPT. MANAGER
ผู้จัดการฝ่าย

รายงานเชิงลึกของแผนก (ขาย/คลัง/จัดซื้อ) แบบสรุปและละเอียด

BUSINESS ANALYST
นักวิเคราะห์

ดึงข้อมูล, สร้างรายงานเฉพาะกิจ, วิเคราะห์แนวโน้ม (กราฟ/สรุป/ละเอียด)

วัตถุประสงค์ของเฟส: แยกภาระการอ่าน (read) ออกจากฐานข้อมูล transactional ด้วย CQRS — report-bc มีฐานข้อมูลของตัวเองที่ denormalized มาแล้ว ทำให้ Dashboard ตอบเร็วโดยไม่กระทบระบบหลัก

Core Use Cases & Data Entry · สร้าง Read Model ยังไง

1

Subscribe events จากทุกบริการ (so.confirmed, stock.deducted, profit.calculated, expiry.approaching ฯลฯ)

2

Project เข้า Read Model — UPSERT เข้าตารางสรุป เช่น sales_summary, profit_by_lot, low_stock

3

Dashboard/API อ่านจาก Read Model โดยตรง (pre-computed) — query เร็ว < 200ms

4

Drill-down รายงานแบบ กราฟ / สรุป / ละเอียด ตามรูปแบบเดิมของระบบ

รายงานหลัก: ยอดขายรายวัน/เดือน, กำไรขั้นต้นต่อล็อต, สินค้าขายดี, สินค้าใกล้หมด (low stock), ล็อตใกล้หมดอายุ, ลูกค้าแต่ละจังหวัด

Edge Cases · กรณีขอบเขต

Event มาไม่เรียงลำดับ

out-of-order → ใช้ version / event timestamp และ UPSERT idempotent เพื่อไม่ให้ข้อมูลเก่าทับข้อมูลใหม่

Event หาย / rebuild

Read model ต้อง สร้างใหม่ได้ (replay) จาก event store/outbox โดยไม่กระทบระบบหลัก

Eventual consistency

Dashboard อาจ lag ไม่กี่วินาที — ยอมรับได้สำหรับรายงาน แต่ต้องระบุเวลา last-updated

Event ซ้ำ

consumer ต้อง idempotent ผ่านตาราง processed_events (event_id unique)

05

Developer Implementation Guide · คำสั่งจาก SA

RULE · DENORMALIZE FOR READ

Read model ออกแบบเพื่อ อ่านเร็ว — ยอมให้ข้อมูลซ้ำ/denormalized เก็บชื่อสินค้า (JSONB) มาด้วย ไม่ต้อง join ข้ามบริการตอน query

RULE · IDEMPOTENT PROJECTION

ทุก consumer ต้อง idempotent ผ่านตาราง processed_events (event_id unique) กันการนับซ้ำ

RULE · HANDLE OUT-OF-ORDER

ใช้ version / updated_at + UPSERT เพื่อกัน event เก่าทับใหม่

RULE · NO SYNC CALL ON QUERY

ตอน query dashboard ห้ามเรียก transactional service — ต้อง pre-compute ล่วงหน้าเท่านั้น

RULE · SEPARATE READ DB

report-bc มีฐานข้อมูลของตัวเอง แยกจากบริการอื่น สามารถ rebuild ได้จาก replay

RULE · EVENT-DRIVEN REFRESH

อัปเดต summary ตาม event ที่เข้ามา ไม่ใช้ polling · ใช้ index ครบเพื่อให้ query < 200ms

RULE · SHOW FRESHNESS

แสดง last_updated บน dashboard เพื่อสื่อสาร eventual consistency ให้ผู้ใช้ทราบ
